制定專案的大腦

Context Files (AGENTS.md) 的威力

🎯 課程目標

大家是不是每次開啟新對話，都要重新交代「我是用 .NET 8」、「我的命名習慣是駝峰式」？今天我們要將這一切自動化。

  • 建立專案大腦： 學習如何藉由 Context Files (AGENTS.md) 讓 AI 自動遵循專案的全局規範。
  • 確保行為高度一致： 建立統一的標準，讓團隊中的每個人（不論是人類或 AI）產出的代碼風格高度統一。
  • 掌握區分 Context 心法： 釐清哪些資訊該寫入靜態設定檔，哪些資訊該由 Agent 動態查詢。

📖 為什麼需要 AGENTS.md？

沒有全域規範的 AI，就像第一天報到的外包人員。他可能懂程式語言，但對你們專案的特殊架構與潛規則一無所知。

  • 減少重複指令 (Prompt Redundancy)： 告別每次開發都要手動複製貼上「技術堆疊說明」與「專案排版準則」的低效做法。
  • 自動載入的預設行為準則： 現代編輯器（如 Cursor）能自動讀取專案根目錄的 `AGENTS.md`，在每一次的 Agent 思考週期中作為全局背景約束。
  • 固化團隊架構決策： 將資深工程師的架構決定以條列式規範固定下來，轉化為 AI 的代碼生成限制，防止初學者帶入不良架構。

💡 頓悟時刻

這是一次思維的翻轉：我們不再試圖去糾正 AI 的每次語法，而是當一個架構師，直接用設定檔去約束它的行為邊界。

Senior 經驗的無痛複製：
將專案規範寫進 AGENTS.md，等同於把資深工程師的架構決定「硬編碼」給每一個協作的 AI。

當專案結構被清晰定義在系統設定檔時，AI 在生成代碼的當下就會主動避開架構地雷，而不需要在 PR 階段才痛苦地要求重構。

🛠️ 實戰：靜態與動態 Context

記住，AGENTS.md 是「靜態」的，寫死架構規則。不要把「動態」的商業邏輯寫進去，動態邏輯應該讓 Agent 自己用工具去查。

  • 靜態 Context (AGENTS.md)：
    • 專門存放「不常變動的技術決策、命名風格、目錄規範、防呆與錯誤處理準則」。
  • 動態 Context (RAG & Tooling)：
    • 動態的資料（如特定的 class 定義、商業邏輯變數、資料庫 Schema）交給 Agent 用 `view_file` 或 `grep_search` 等工具即時查找，避免寫死在設定檔中造成 Token 爆滿。
# AGENTS.md 範例
## Backend 規範
1. 所有 API 回傳值必須包裝在 ApiResponse<T> 中
2. 嚴禁 Controller 直接呼叫 DB 實體，必須透過 Service 層
3. 變數與欄位命名一律使用小駝峰式 (camelCase)

⚠️ 常見誤區

AGENTS.md 也是專案程式碼的一部分，它需要經歷嚴格的 Code Review。它寫爛了，整個專案生成的 Code 都會跟著爛。

  • 把設定檔當成垃圾桶：
    • 往裡面塞入大量暫時性的除錯指令、甚至是具體的實作程式碼，導致 Token 消耗劇增，且 AI 反而容易混淆。應保持設定檔的精簡與權威性。
  • 忘記維護與更新：
    • 專案架構已經升級到 .NET 9，但 `AGENTS.md` 還留在 .NET 8，導致 AI 持續生成舊架構代碼。
  • 最佳實踐： 將 `AGENTS.md` 納入 git 版本控管，只要專案架構有調整，「第一件事就是去更新 AGENTS.md」。
