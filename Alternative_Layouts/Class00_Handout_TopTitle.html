🎯 為什麼有些工程師離不開 AI，而有些還覺得沒用？

這是一個觀念的轉折點。在進入實作前，請先問問自己，以下哪個觀點你認同？

  • AI 會犯錯： 所以還不能信任，每次生成的程式碼都要大量手動修正。
  • AI 輸出不穩定： LLM 非確定性的特質使得產出結果像在碰運氣。
  • AI 就是聊天： 輸入中文拿到答案，不需要特別去學習專門的系統架構。
  • 想 Prompt 太花時間： 撰寫詳細的 Prompt 往往比自己直接寫程式碼還要慢。
🔑 核心觀念：目前語言模型的能力已經足夠，而你的使用方式與心智模型將直接決定產出結果。

🎯 誤解一：AI 會犯錯，所以還不能信任

許多開發者因為 LLM 出現幻覺或錯誤就感到失望。但我們需要理解，大型語言模型的本質就是「機率分佈」，而非確定性的邏輯處理器。

真正的關鍵問題不是「AI 會不會錯？」，而是「當它出錯的時候，你有沒有健全的驗證機制去發現？」。

在軟體開發中，我們早就習慣處理各種不確定性與誤差（例如浮點數精度誤差、網路封包不穩定、使用者的垃圾輸入等），我們從不因此放棄使用這些工具。關鍵在於建立起對應的「異常處理與驗證機制」，這也是我們後續課程的重點。

🎯 誤解二：AI 非確定性，所以結果靠運氣

「Input ➔ Output 永遠一致」是我們對傳統程式碼的認知。但如果我們把需求丟給三個資深工程師，也會得到三種截然不同的解法，這不代表失控，只代表解決問題的路徑不只一條。

真正影響 AI 產出品質的通常不是運氣，而是我們給予它的引導框架：

  • Context 工程： 是否給予了足夠、精準的專案上下文資訊。
  • 明確的目標定義： 任務是否被拆解得夠細、邊界是否足夠清晰。
  • 可行的驗證機制： 是否有讓 AI 可以進行自我糾錯的測試案例或檢核點。

🎯 誤解三：AI 就是聊天，不需要學

這是最危險的誤解。因為 AI 的介面極其簡單（對話框），導致許多人誤以為不需要學習就能完全發揮它的能力。

如果真的只是聊天，為什麼同一個模型，在某些人手中能帶來數倍的效率提升，而在另一些人手中卻只能輸出無法運行的廢扣？

差異不在模型的能力，而在使用者引導模型的工作流。

在接下來的課程中，我們將跨越簡單的 Prompt Engineering（對話調教），深入學習 Context Engineering（上下文工程） 以及如何建構 Agentic Workflow。

🎯 誤解四：想 Prompt 比自己做還慢

這個說法背後隱含了一個錯誤的假設：「我們應該用跟人類一模一樣的工作模式來對待 AI」。如果你找一個自己已經知曉標準答案的問題考 AI，那確實是多此一舉。

事實上，AI 與人類的擅長領域完全不同，它在以下場景具有壓倒性的效率優勢：

  • 大量閱讀與整理： 在數秒內讀完幾百頁文件並提取出關鍵的實作範本。
  • 重複與繁瑣工作： 生成大量的 Unit Test 骨架、DTO 轉換或樣板程式碼。
  • 跨領域快速搜尋： 快速找出特定套件在不同版本間的 API 變更。

將 AI 放在對的定位上，才能真正釋放它的產能。

🎯 總結：兩種 AI 心智模型

在進入後續的實戰課程前，請試著翻轉你的思維模式，將心智模型從舊有的模式升級為工程思維：

  • 舊心智模型 (對話者)： 將 AI 視為一個用來問答的聊天機器人。它回答什麼我就複製什麼，遇到錯誤就繼續盲目追問。
  • 新心智模型 (工程系統)： 將 AI 視為一個能力極強但需要清晰約束、精準上下文與驗證環節的軟體工程系統。
如果你只把 AI 當成聊天機器人，你將只能得到聊天機器人等級的瑣碎價值；但如果你把 AI 當作軟體工程工具來管理與引導，它將為你帶來生產力上的革命性突破。
