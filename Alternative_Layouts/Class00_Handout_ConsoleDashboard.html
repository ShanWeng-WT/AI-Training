開場：你如何看待當下的 AI 工具？

許多工程師試用過 AI 後，很快又退回純手寫模式。這通常不是因為工具不夠強，而是我們將舊的心智模型套用在新世代的工具上。

🔑 核心觀念：模型能力已足夠 ➔ 你的使用方式與工作流，決定產出結果的品質。

誤解一：AI 會犯錯，所以還不能信任

LLM 的底層邏輯是「統計機率的文字預測」，而非嚴謹的邏輯引擎。因此，它一定會出錯。

真正的問題從來不是「AI 會不會錯？」
而是「當它出錯時，你的工作流有沒有能力捕捉並更正它？」

我們日常開發早就習慣了浮點數不精準、網路斷線等異常。對待 AI，我們也需要建立相應的例外捕捉與驗證機制。

AiVerificationWorkflow.cs 
// 概念化驗證防禦機制
try 
{
    var generatedCode = await AI.GenerateCodeAsync(context);
    
    // 關鍵在於：建立對應的驗證屏障
    CodeValidator.AssertNoHalucinatedApi(generatedCode);
    CodeValidator.AssertFollowsProjectNamingStyle(generatedCode);
} 
catch (ValidationException ex) 
{
    Logger.LogWarning("偵測到 AI 輸出偏離規範，自動退回重新修正...");
}

誤解二：AI 非確定性，所以結果靠運氣

工程師習慣「給予固定輸入，得到唯一輸出」。但 LLM 更像是一個動態的機率場，產出非唯一解，但可以被高度收斂。

就像請三位資深工程師寫同一個功能，會有三種不同寫法。這不是失控，而是解決路徑的多樣性。

收斂機率的黃金鐵三角： 完美的 Context、明確的 Constraint 與防呆驗證。

probability_shaping.yml 
# 透過 Context 與 Constraints 將機率整形 (Probability Shaping)
probability_收斂機制:
  context_上下文: "提供專案既有程式碼規範 + 資料庫綱要"
  target_明確目標: "只負責生成 Repository 層，其餘禁止"
  constraints_約束:
    - "禁止直接使用 DbContext"
    - "必須回傳 Task"
  validation_自我校正: "自我檢查是否使用了 API 捏造"

誤解三：AI 就是聊天，不需要學

簡單的輸入介面隱藏了極其複雜的模型運作。許多人覺得 AI 難用，是因為他們只把它當「點餐系統」，而非「協作系統」。

差異不在模型，而在使用者。

若要發揮完全效率，我們需要跨越 Prompt 工程，進入以專案感知為中心的 Context Engineering（上下文工程）。

context_engineering_manifesto.md 
# Context Engineering 核心法則
1. **主動餵入 (Feed)**：餵入正確、相關的實體介面與程式碼片段。
2. **剔除干擾 (Prune)**：將非必要的程式碼排除，提高訊噪比。
3. **黃金範本 (Few-shot)**：給予 1~2 個符合企業最高標準的
   完美程式碼範例，強制 AI 進行模式匹配。

誤解四：想 Prompt 比自己做還慢

如果您拿 AI 來寫一行簡單的 Lambda 表達式，那確實比自己動手慢。因為您用錯了地方。

AI 特別擅長處理：閱讀、搜尋、大量整理、重複工作與框架轉換。

把重複性、高阻力的「體力活」交給 AI，把創造性、設計架構與關鍵決策留在您的手中。

DeveloperVsAiWorkload.cs 
// 合理的工作分工
public class TaskDistribution 
{
    // 人類工程師的核心職責：架構與關鍵邏輯
    public void DesignArchitecture() => Humans.DecideDesignPatterns();
    
    // AI 助手的核心職責：高重複性與體力活
    public async Task HandleTediousLaborAsync() 
    {
        await AI.GenerateDtoClassesAsync();
        await AI.GenerateUnitTestingBoilerplateAsync();
        await AI.ReadAndSummarizeThirdPartyDocsAsync();
    }
}

總結：兩種 AI 心智模型

🤖 舊心智模型 (對話者)

定義：將 AI 當作一個隨意問答的聊天機器人。

  • 沒有明確的工作流程與約束
  • 完全依賴模型本身的零星智商
  • 遇到幻覺或報錯，只能無腦追問
  • 產出價值僅限於簡答與參考

⚙️ 新心智模型 (工程系統)

定義：將 AI 當作一個需要管理與引導的軟體工程系統。

  • 提供精確的 Context 與約束條件
  • 建立穩健的 Try-Catch 自我校正
  • 使用黃金範本進行模式收斂
  • 產出高度符合企業開發規範的代碼

💡 核心行動法則

要點：

  • 不要聊天： 使用有結構的 Agent Skills 定義任務。
  • 工具定位： 將 AI 作為您工作流中的「初級工程師」，您是架構師。
  • 建立信心： 透過在系統中編寫防呆與驗證，建立人機協作的安全網。
