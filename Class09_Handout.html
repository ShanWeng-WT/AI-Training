Spec 驅動開發 (SDD)

為何需要 Spec 驅動開發 (SDD)？

當開發規模擴大，模糊的需求會導致 AI 產生高耦合、無法編譯的程式碼。我們必須從「打字機」轉變為「架構審閱者」。Spec-Driven Development (SDD) 的核心，是在動工前先以一份結構化的規格書（Spec）凝聚共識。

🚨 大型任務的痛點：

  • 義大利麵條程式碼： AI 容易在無規劃下產生數百行難以 Debug 的高耦合程式碼。
  • 除錯成本高昂： 後續進行 Debug 與調整結構的時間往往比自己重寫還要多。
  • 主導權喪失： 工程師淪為拼湊程式碼的工具，而非架構的掌控者。
🔑 核心轉變：從「程式碼撰寫者」轉變為「架構審閱者」，先以 Spec 鎖定架構，再產出程式碼。

撰寫精準的 Spec

將 AI 視為新進的資深工程師。為其提供明確的 Onboarding 指南，劃定明確的運作邊界。

📋 撰寫 Spec 的五大核心要素：

  • Context (現況與痛點)： 描述目前系統的問題與背景。
  • Goals (預期功能)： 明確定義此重構或開發必須達到的目標。
  • Non-Goals (非目標)： 絕對不要做的事（如：先不要調整 UI 或資料庫），防止開發範圍失控（Scope Creep）。
  • Constraints (限制條件)： 套件版本、平台限制（例如 Unity 版本或 .NET 限制）。
  • Architecture Rules (設計模式與規範)： 規定所採用的設計模式與程式碼風格。
💡 最重要的關鍵在於「Non-Goals (非目標)」，它能幫助 AI 聚焦並避免不必要的衍生修改。

分步施工與驗證：Spec → 分步施工 → 驗證

有了 Spec 並不代表要 AI 一次寫完。SDD 的精髓在於把規格拆成有序的施工步驟，每完成一步就 Build / Test 驗證，確認無誤再進入下一步。

  • 分步施工： 在 Spec 中以 Steps 標明執行順序，AI 一次只動工一個步驟。
  • 即時驗證： 每步完成後立即 Build 與跑測試，把錯誤鎖在最小範圍內。
  • 漸進式重構： 驗證通過才前進下一步，避免一次性大改造帶來的失控風險。
💡 一份好的 Spec 不只描述「做什麼」，也定義「依什麼順序做、如何驗收」。（選用哪個模式已在 Class 04 介紹）
Spec.md 
## Steps (依序施工，逐步驗證)

1. 抽出 IInventoryService 介面
   - Verify: 專案可編譯 (Build OK)

2. 實作 InventoryService 並注入
   - Verify: 既有 EditMode 測試全綠

3. 將 UI 改為呼叫介面
   - Verify: PlayMode 拾取道具流程正常

## Non-Goals
- 先不調整存檔格式與 UI 樣式

HITL 架構審閱

將人類工程師的經驗融入開發循環 (Human-in-the-loop)。在 AI 動工前，嚴格把關實作藍圖的合理性。

🔍 架構審閱 Checklists：

  • 耦合度檢查： 模組之間的依賴關係是否合理？是否會引入不必要的相依性？
  • 破壞性變更： 是否會破壞既有的 Public APIs 或導致現有系統異常？
  • 效能地雷： 是否存在 GC 效能問題或潛在的資源洩漏？
  • 資源與生命週期： 在 Unity 中，事件（Events）是否確實解除綁定（Unsubscribe）？
💡 發現任何不合理，要求 AI 「更新 Spec」並修正 Markdown。直到計畫完美，才回覆 Approve 開始施工。

常見陷阱與防禦

在使用 AI 處理大型架構時，需要建立防禦機制以防失控與幻覺。

🛡️ 核心防禦三部曲：

  • 避免盲目接受： 強制 AI 一次只執行一個步驟 (Step-by-step)，完成後由人工驗證再繼續。
  • 拆分大任務： 避免一口吃成胖子。若 Spec 過於龐大，應將其拆分成多個子 Spec，進行漸進式重構。
  • 防範忽略邊界條件： 必須在 Spec 中明文規定例外處理機制與錯誤回傳規範，防止 AI 忽略異常情境。
🔑 記住：修改 Markdown 計畫只需幾秒鐘，但 Debug 錯亂的程式碼卻要花上數小時。
