無痛實作大架構

Spec 驅動與 Planning Mode (PlanMode)

一、為何需要 Spec 驅動與 PlanMode？

  • 傳統痛點： 模糊需求會產生義大利麵條程式碼 (Spaghetti Code)，高耦合、難維護。
  • 角色轉變： 開發者不再只是「打字機」，而是「架構審閱者 (Architecture Reviewer)」。
  • 解決方案： 透過精準的 Spec 與 PlanMode 產生實作計畫，審閱後再產出程式碼。

二、撰寫精準的 Spec (Specification)

框定 AI 的邊界與限制，避免 Scope Creep。

  • Context (背景)： 痛點與現況。
  • Goals (目標)： 預期達成的功能。
  • Non-Goals (非目標)： 明確告知「不」該做什麼！
  • Constraints (限制)： 引擎版本、相容性、套件限制。
  • Architecture Rules (規範)： SOLID、DI、EventBus 等設計模式。
💡 專家提示： 視 AI 為剛到職的資深工程師，Spec 是他的 Onboarding 說明書。

三、Planning Mode (PlanMode) 核心精神

「先產出實作步驟與架構設計，別急著寫程式碼。」

  • 目錄結構異動： 新增、刪除或修改的檔案路徑。
  • 架構依賴： 模組互動與關聯。
  • 分步執行： 將實作拆分為 Step 1, Step 2... 逐步進行。
# Prompt: 啟動 PlanMode
接下來我們將進行架構重構。請進入 Planning Mode (PlanMode)。
請根據 Spec，**不要**直接輸出程式碼，先產出一份 `ImplementationPlan.md`。
包含：檔案異動、架構設計、執行步驟。
當我回覆 "Approve" 後，再逐步執行。

四、Human-In-The-Loop (HITL) 架構審閱

這是工程師發揮最大價值的地方。修正 Markdown 遠比 Debug 程式碼快！

  • 耦合度檢查： 是否有不合理的依賴？
  • 破壞性變更： 是否修改了公用介面 (Public APIs)？
  • 效能考量： Update 內是否有 `GetComponent`？
  • 資源管理： 離開場景時是否確保 Unsubscribe 事件？

防禦機制： 發現問題，請要求 AI 「更新 Implementation Plan」，而不是直接動手修 Code。

五、常見陷阱與防禦 (Common Pitfalls)

  • 盲目接受 (Blind Acceptance)： 未審閱就喊 Go。解法： 強制 AI 一次只執行一個 Step。
  • Big Bang Approach： Spec 過於龐大導致 AI 幻覺。解法： 漸進式重構 (Strangler Fig Pattern)。
  • 忽略邊界條件： 忘記 Null 檢查與例外處理。解法： 明文規定於 Architecture Rules 中。
