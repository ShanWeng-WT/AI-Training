Spec 驅動與 Planning Mode (PlanMode)

一、為何需要 Spec 驅動與 PlanMode？

  • 模糊需求 = 義大利麵條程式碼
  • 你不是打字機，你是架構審閱者
  • 先有計畫，再有程式碼
告別高耦合，掌握主導權。

二、撰寫精準的 Spec

  • Context： 現況與痛點
  • Goals： 預期功能
  • Non-Goals： 絕對不要做的事
  • Constraints： 版本與套件限制
  • Architecture Rules： 設計模式與規範
把 AI 當新進資深工程師，Spec 就是他的 Onboarding 指南。

三、PlanMode 核心精神

「先產出實作步驟與架構設計，別急著寫程式碼。」
  • 確認異動： 檔案路徑與變更
  • 釐清依賴： 模組互動關係
  • 分步執行： 拆分 Step 逐步進行
# 啟動 PlanMode
進入 Planning Mode。不要寫 Code。
先產出 ImplementationPlan.md（含檔案異動、架構、步驟）。
我回覆 Approve 後，再分步執行。

四、HITL 架構審閱

  • 耦合檢查： 依賴是否合理？
  • 破壞變更： 影響 Public APIs？
  • 效能考量： 藏有耗能語法？
  • 資源管理： 確實解除綁定？
修正 Markdown，遠比 Debug 程式碼快！
發現問題，要求 AI 「更新 Plan」。

五、常見陷阱與防禦

  • 盲目接受： 強制 AI 一次只跑一 Step。
  • 一口吃成胖子： Spec 拆分，漸進式重構。
  • 忽略邊界條件： 在 Spec 明文規定例外處理。
