# Data Usage Analysis (Server / Client / Unity Editor)

## What this produces

A markdown table classifying every exported data table by where it is actually
**referenced at runtime**, plus a short findings section. Example shape:

```
| Data | Server | Client | Unity Editor |
|------|--------|--------|--------------|
| FishSetting | O | O | O |
| FishScaleAttrData | O | - | O |
| FishAttrData | - | O | O |
```

The point is to separate three independent questions for each data table:
1. Does the **server** (the `Modules\...` C# code) read this file at runtime?
2. Does the **client runtime** (non-Editor scripts in `Model.Unity\Assets`) read it?
3. Is the file only used by **Unity Editor** tooling (editor windows, inspectors,
   data generators) and never loaded at client runtime?

## Symbol rules (apply consistently)

For each of the three columns:

| Symbol | Meaning |
|--------|---------|
| **O** | Confirmed referenced at runtime on that side (found a concrete code/asset reference). |
| **X** | The file **exists** in that side's path, but no reference was found — confirmed unused. |
| **-** | The file does **not exist** in that side's path at all (Not Applicable). |
| **?** | Could not determine with confidence — **flag for human review** (e.g. an indirect/dynamic reference, a name built at runtime, or an ambiguous match). |

Key consequences:
- The **Server** column is `-` whenever the table has no file under the server
  CustomData folder. It is `O`/`X`/`?` only when a server file exists.
- The **Client** column is `-` whenever the table has no file under the client
  Unity `Data` folder.
- The **Unity Editor** column is `-` only when the file is not present anywhere in
  the Unity Assets tree. Because the client `Data` folder lives inside Unity, a
  file present there is in-scope for the Unity Editor column (`O` if editor tools
  reference it, `X` if nothing in Unity references it).
- Annotate a generated file with `O (產出)` when the file is produced as **output**
  of a Unity Editor tool (the tool writes/saves the file) rather than hand-authored.
  If the file is generated by a Unity Editor tool but is NOT saved in the Unity Data 
  folder (e.g. exported directly to the server side), mark the Unity Editor column 
  as `- (產出)` or `X (產出)` to indicate it is produced by the editor but doesn't exist 
  in the client runtime path. If you cannot tell whether it is generated, use `O` 
  and note the uncertainty.

Prefer `?` over guessing. A wrong `O`/`X` is worse than an honest `?`.

## Step 1 — Resolve the roots (never hard-code project numbers)

Project numbers vary (`Game190` vs `Game###`, `FishHunter4` vs `FishHunter#`).
Resolve them from the workspace root by pattern:

```pwsh
pwsh -File <skill>/scripts/Resolve-Roots.ps1 -WorkspaceRoot <workspace>
```

This returns JSON with `ServerCustomData`, `ClientDataRoot`, `ClientAssetsRoot`,
`ServerCodeRoot` (`Modules`), `FishModuleRoot`, `GameFolder`, `FishModuleName`.
If any value is `null`, stop and ask the user for that path — do not guess.

## Step 2 — Build the data-entity inventory

```pwsh
pwsh -File <skill>/scripts/Build-DataInventory.ps1 `
  -ServerCustomData <ServerCustomData> -ClientData <ClientDataRoot>
```

This returns the canonical list of data entities (one per data table, wave-family
files `WaveKind*` / `WaveBonusKind*` collapsed into a single `WaveKindXXX`), each
with `InServer` / `InClient` booleans and the concrete `ServerFiles` /
`ClientFiles` names. The booleans already fix every `-` cell. Reference search
only has to decide the remaining cells.

## Step 3 — Dispatch two analysis subagents in parallel

Run a **server** subagent and a **client** subagent in the same turn (one message,
two task calls). Give each one: the resolved roots, the full entity inventory, the
symbol rules above, and the search guidance below. Each must return, per entity, a
verdict **plus the evidence** (file:line of the reference, or "no reference found").

### Server subagent task

Goal: fill the **Server** column for every entity whose `InServer` is true.

- Search the server code only: `ServerCodeRoot` (`Modules`), concentrating on
  `FishModuleRoot` (e.g. `...\FishHunter<N>\...`). The central loader is
  `GameModel.Internal\RoomHelper.cs`, which reads files via
  `File.ReadAllText(Path.Combine(dataPath, "<FileName>"))` and a `FileName`
  constants class — but references may be spread across the module.
- For each entity, search for: the exact filename(s) from `ServerFiles` (with
  extension), the bare entity name, and any `FileName.<Entity>` constant. Use
  ripgrep over `*.cs`.
- Verdict: a real read/parse reference → `O`; file exists but zero references → `X`;
  only a dynamic/indirect/ambiguous reference → `?`.
- Ignore matches that are merely class/type names unrelated to loading the file,
  and ignore test-only or commented-out code (note it if unsure → `?`).

### Client subagent task

Goal: fill the **Client** and **Unity Editor** columns for every entity whose
`InClient` is true (and check the Unity Editor column for any entity whose file
appears anywhere under `ClientAssetsRoot`).

- Search client code/assets under `ClientAssetsRoot` (`Model.Unity\Assets`).
- **Classify each reference by location**:
  - A reference in a **runtime** script (NOT under any `Editor/` folder, and not in
    an editor-only assembly) → contributes to **Client = O**.
  - A reference only in an **Editor** script (path contains `\Editor\`, or the file
    is in an editor asmdef) → contributes to **Unity Editor = O**.
- Also consider non-script references: a data file referenced by a scene/prefab via
  its `.meta` GUID, or loaded by `Resources.Load` / `TextAsset` / `Addressables`,
  counts as a runtime reference. (Searching the bare entity name plus the concrete
  filename across `*.cs` usually suffices; escalate to GUID search only if a file
  looks unused but you suspect a scene/prefab binding.)
- **Note on Dev Tools**: Scripts that act as dev-only simulation tools (such as simulator scenes or testing scripts) 
  should be considered Unity Editor tooling, not Client runtime. Files referenced ONLY by dev tools 
  should be `X` in the Client column, not `O` or `?`. Watch out for file loading that relies on hardcoded `Assets/...` paths, which usually indicate dev-only usage.
- **Generated `(產出)` detection**: if an Editor tool *writes/saves* the file (look
  for `File.WriteAll*`, `StreamWriter`, `AssetDatabase.CreateAsset`, `SaveAsset`,
  export/generate menu items targeting this file), mark Unity Editor as `O (產出)`.
- Verdicts: runtime reference found → Client `O`; file in client `Data` but no
  runtime reference → Client `X`; dynamic/ambiguous → Client `?`. Same logic for the
  Unity Editor column using editor-scope references.

Each subagent returns a table (Entity → verdict + evidence). Keep evidence concise
(`file:line` is enough) so it can be shown to the user if they want to spot-check.

## Step 4 — Merge and present

1. Combine the two subagents' verdicts with the inventory's `-` cells into one
   table, entities sorted alphabetically (with `WaveKindXXX` as one row).
2. Generate a markdown file named `Game###-data-analysis-result-{date}.md` to the 
   user's current directory (or any path/name the user specifies).
3. This generated file must include:
   - The result table (using the exact output template below).
   - Unconfirmed findings or noteworthy patterns (e.g., generated-vs-authored notes, files present but completely unused).
   - The file paths of the code or assets that reference the data (from the subagents' evidence), to serve as an audit trail.

### Output template

```markdown
# Data Usage Analysis — <project / GameFolder>

## 欄位及符號說明
- **Server**：server 執行時會使用到。
- **Client**：client 執行時會使用到。
- **Unity Editor**：檔案放在 Unity 專案內，但 Runtime 不會用到，通常是透過 Unity Editor 工具產出/使用的資料。
- **O** 需要用到　**X** 不需要用到　**-** 該路徑下不存在此檔案 (N/A)　**?** 無法確定，需人工確認

| Data | Server | Client | Unity Editor |
|------|--------|--------|--------------|
| ... | ... | ... | ... |

## Findings
- ...
```

## Notes & limits

- This skill judges usage by **references in code/assets**, not by the Designer's
  Doc/VBA export rules. A file can be deployed but unreferenced (→ `X`), which is a
  useful signal but not proof it is safe to delete — call that out, don't assert it.
- Dynamic filename construction (names built from variables/loops, e.g. wave files
  selected by index) is the main source of `?`. When a whole family is loaded by a
  pattern, treat the family (`WaveKindXXX`) as referenced if the loading pattern is
  found, and say so in Findings.
- Prefer `rg` (ripgrep) for searches; it is fast over these large trees.
