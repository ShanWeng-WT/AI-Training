第 10 堂：超越單次對話

Agentic Workflow 設計模式

一、為何需要 Agentic Workflow？

  • 單次對話瓶頸： 容錯率極低，容易在幾百行程式碼中出錯，缺乏反覆測試驗證機制。
  • Agentic 精神： 讓 AI 像人類工程師一樣工作。給予反思、使用工具、拆解任務與協作的能力。
  • 高複雜度應對： 完善的 Workflow 能讓 GPT-3.5 表現超越單次提問的 GPT-4。

二、模式一：反思 (Reflection)

「讓 AI 睜開眼睛看錯誤訊息，自我修正。」

  • 機制： 腳本自動擷取編譯器或靜態分析工具的報錯。
  • 反饋： 將錯誤自動餵回給 AI，要求其反思並修復問題。
  • 效益： 工程師最終拿到的程式碼即是「已通過編譯」的版本。

三、模式二：工具使用 (Tool Use / Function Calling)

「賦予 AI 存取真實世界資料的手與眼。」

  • 突破限制： AI 無法直接通靈資料庫狀態，提供 JSON 格式的 API 工具描述。
  • 運作流程： AI 判斷需要資訊 ➔ 發送函數呼叫請求 ➔ .NET 伺服器執行真實 SQL 查詢 ➔ 結果回傳 AI ➔ 總結答案。
  • 應用場景： 客服機器人查詢玩家裝備、AI 查詢專案特定文件內容。

四、模式三：規劃 (Planning)

「先思考再行動，拒絕大雜燴式的輸出。」

  • 理論基礎： Chain of Thought (CoT) 思維鏈。
  • 拆解步驟： 面對複雜任務，要求 AI 先輸出實作藍圖（Step 1, Step 2...）。
  • 逐一擊破： 透過腳本嚴格限制 AI 每次只執行一個 Step，降低邏輯跳躍與幻覺風險。

五、模式四：多代理協作 (Multi-agent Collaboration)

「組建虛擬 AI 開發團隊，互相批判與合作。」

  • 分工合作： 實例化多個具備不同 System Prompt 的 Agent（如 Coder, Tech Lead, QA）。
  • 交互審查： Coder 產出程式碼 ➔ Tech Lead 嚴格審查效能 ➔ 發現問題退回重寫。
  • 維持專注： 單一 Agent 只需專注於特定角色任務，避免上下文失焦。

六、陷阱與防禦機制

  • 無窮迴圈： Reflection 一直修不好。解法： 設定最大迭代次數 (Max Iterations)。
  • 過度自信幻覺： 捏造 API 參數。解法： 強型別驗證，溫和報錯回傳讓 AI 修正。
  • 危險工具操作： 呼叫 `DROP TABLE`。解法： 高風險操作強制啟用 Human-in-the-loop 人類確認。
