超越單次對話

Agentic Workflow 設計模式與 Hooks 自動化護欄

為何需要 Agentic Workflow？

單次對話的容錯率極低。引進 Agentic Workflow 能讓 AI 具備自我修正、工具使用與分工協作的能力，大幅提升複雜任務的成功率。

💡 吳恩達 (Andrew Ng) 的核心論點：

  • 超越模型本身的限制： 良好的設計工作流甚至能讓 GPT-3.5 的表現超越直接進行單次對話的 GPT-4。
  • 像人類工程師般思考： 讓 AI 具備自我 Debug、查找資料、按部就班做計畫，以及團隊分工的邏輯。
  • 容錯率高： 透過迭代運算 (Iterative Loop)，AI 可以不斷修正自己的輸出直至達標。

模式一：反思 (Reflection)

讓 AI 睜開眼睛看錯誤訊息。透過編譯與執行期的錯誤回饋，實現自動化的自我修復。

「不要讓 AI 閉著眼睛寫 Code，而要建立自動化回饋機制。」

🔄 反思工作流的運作步驟：

  • 執行與編譯： 寫出自動化腳本，將 AI 產出的程式碼自動進行編譯或執行。
  • 錯誤擷取： 當編譯失敗或拋出 Exception 時，擷取完整的報錯訊息。
  • 回饋與修復： 將錯誤訊息直接丟回給 AI，要求它分析原因並重新產出，直到通過編譯。

模式二：工具使用 (Tool Use)

賦予 AI 存取外部世界與系統資料的能力。透過 Function Calling，讓 AI 能夠調用 API 取得精確的最新狀態。

「AI 不能通靈，如果你需要它處理即時資料，必須給予它手與眼。」

🛠️ 運作機制 (Function Calling)：

  • 定義描述： 提供 JSON 格式的 API 工具描述與參數架構。
  • 語意分析： AI 分析使用者意圖，判斷是否需要呼叫工具，並產出參數 JSON。
  • 伺服器執行： 系統接收 JSON 並在後端執行 API（如：查詢玩家背包、讀取資料庫）。
  • 結果回傳： 將 API 執行結果餵回給 AI，由其綜合成人類讀得懂的語意回答。

模式三：規劃 (Planning)

先思考再行動。將龐大的需求大卸八塊，透過分步執行降低模型的幻覺風險，保持程式架構乾淨。

「面對複雜任務，拒絕大雜燴式的輸出。」

📋 規劃模式核心實作：

  • 思維鏈 (Chain of Thought)： 要求 AI 寫出推導思路，而非直接給答案。
  • 藍圖拆解： 強迫 AI 在開發前先規劃「實作藍圖」，列出所有子步驟。
  • 單步執行與驗證： 配合自動化腳本，一次只命令 AI 執行一個步驟，完成並通過人工或單元測試後，才推進至下一步。

模式四：多代理協作 (Multi-agent)

組建虛擬 AI 開發團隊。透過明確的角色定義與互相審查，克服單一模型的上下文限制與偏見。

「讓不同專長的 Agent 互相合作與批判，完美複製人類開發管線。」

👥 虛擬開發團隊範例：

  • Coder Agent (編碼人員)： 專注於撰寫程式碼以實現 Spec 的功能需求。
  • Tech Lead Agent (技術主管)： 專注於 Code Review。審查程式碼的效能、擴充性、Unity 的 GC 地雷等，若有問題則退回給 Coder 修改。
  • 優勢： 將複雜邏輯拆分到不同 Prompt 角色中，避免單一 Prompt 因規則過多而失焦。

Hooks 自動化護欄：把驗證變成自我修復的閉環

前面幾堂課的驗證多半是「手動」的：人工讀 diff、手動跑 Build 與 Test。Hooks 的價值，就是把這些手動驗證 (尤其是 Reflection 模式的錯誤回饋) 自動化成一個自我修復的閉環——在 AI 寫入程式碼前/後自動觸發編譯器、Linter 與測試，失敗就把報錯自動回傳給 AI 重試，成功才放行。

🛡️ 兩層防禦：

  • 層級一・引導式自我修正 (文字防禦)： 在 Prompt / Skill 尾端加上虛擬 Code Review 提示，要求 AI 自行複查，是成本最低的第一道防線。
  • 層級二・工具鏈 Hooks 攔截 (工程防禦)： 將 AI 工具與編譯器 / Linter 管道串接，偵測到失敗時自動攔截寫入，並把 Error Logs 送回 AI 重試。
Prompt ➡️ AI 產生代碼 ➡️ Hook 攔截 ➡️ 編譯/測試驗證 ➡️ (失敗) 報錯回傳重試 ➡️ (成功) 寫入專案
「承認 AI 是機率性的，並用確定性的工具 (Hooks) 包圍它！」
hooks.config.yaml 
# PostToolUse Hook：AI 寫檔後自動驗證
hooks:
  - event: post_write           # 攔截「寫入程式碼」這個動作
    matcher: "*.cs"             # 僅針對 C# 檔案觸發
    pipeline:
      - run: dotnet build        # 1. 編譯驗證
      - run: dotnet test         # 2. 單元測試
      - run: dotnet format --verify  # 3. Linter / 風格掃描
    on_failure:
      action: return_to_agent    # 失敗：把 Error Logs 回傳給 AI
      max_retries: 3             # 避免無限重試 (機率性 → 確定性收斂)
      redact:                    # 日誌脫敏：回傳前遮蔽敏感資訊
        - absolute_paths
        - ip_addresses
        - credentials
    on_success:
      action: commit             # 成功：才放行寫入專案

常見陷阱與防禦機制

在使用 Agentic Workflow 與 Hooks 釋放 AI 自動化威力的同時，必須建立嚴格的防護牆以防失控與無限消耗。

🛡️ 企業級防禦手段：

  • 無窮迴圈防禦： 當 AI 自我 Debug 一直失敗時，極易陷入死循環。Hooks 必須設定 Max Retries = 3 強制中斷，避免無限重試燒掉 Token 與時間。
  • 編譯過 ≠ 好代碼： 通過編譯只代表語法正確，不代表品質達標。需在 Hook 管道掛上 Linter 做效能與漏洞掃描，攔截 Unity GC 地雷或安全弱點。
  • 日誌脫敏 (Log Redaction)： 把 Error Logs 回傳給 AI 前，務必遮蔽絕對路徑、IP、憑證等敏感資訊，避免內部機密外洩給雲端模型。
  • 過度自信幻覺： AI 可能虛構 API。需搭配「強型別驗證」與「溫和報錯回傳」，讓 API 回傳清晰的錯誤以引導其修正。
  • 高風險操作防禦： 針對修改資料庫、刪除系統檔案等危險操作，必須加入 Human-in-the-loop，設計確認機制才能放行。
