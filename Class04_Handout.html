選對大腦：模型分級、Thinking Budget 與 Plan Mode

企業內部 AI 輔助開發教育訓練

🎯 課程目標

在你按下「送出」之前，其實已經悄悄做了三個關鍵抉擇——選哪個模型、給多少思考預算、用哪個模式。多數人對這三件事毫無意識，任務的成本、速度與品質卻早已被它們決定。這堂課，就是要教你把這三個選擇做對。

  • 選對大腦 (Model Tiering)： 學會依任務難度配對「剛好夠用」的模型，在成本、延遲與推理品質之間做出工程級的取捨。
  • 調好預算 (Thinking Budget)： 理解「延伸思考」的本質，掌握何時加大、何時關小思考預算的心法。
  • 選對模式 (Plan Mode)： 認識「先規劃、再動手」的模式選擇，在大型任務中避免 AI 直接噴出失控的程式碼。
💡 本堂主軸： 聚焦「送出 User Prompt 之前的所有抉擇」，讓你在開口之前，就先贏在起跑點。

🧠 模型分級 (Model Tiering)

沒有「最好」的模型，只有「最適合這個任務」的模型。請依任務的複雜度與風險，挑選對應等級的大腦——別用牛刀殺雞，也別拿鈍刀砍硬骨頭。（以下以 Claude 家族為例，但原則通用於所有模型供應商。）

⚡ 輕量快速 (如 Haiku)

反應快、成本低，適合大量、低風險、答案明確的任務。

  • 格式轉換、資料清洗、欄位對映
  • 批次摘要、分類、標籤化
  • 簡單樣板程式碼產生

取捨： 省錢省時，但複雜推理會力不從心。

⚖️ 中階均衡 (如 Sonnet)

速度與推理的甜蜜點，是日常開發的主力大腦。

  • 一般功能開發與重構
  • 常規 Bug 修復與測試撰寫
  • 程式碼審查與文件撰寫

取捨： 多數任務的預設首選，CP 值最高。

🚀 旗艦推理 (如 Opus)

推理品質最強，專為艱難任務而生，成本與延遲也最高。

  • 複雜系統架構設計
  • 棘手、跨模組的深層 Debug
  • 多步演算法與權衡分析

取捨： 品質換成本，只在真正需要深度推理時出動。

💭 Thinking Budget (思考預算)

「延伸思考 (Extended Thinking)」是給模型一段額外的 token 預算，讓它在正式回答前先「想清楚」——拆解問題、嘗試推理、自我檢查。這就像考試前允許學生在草稿紙上演算，正確率會明顯提升。但天下沒有白吃的午餐：更多思考 = 更高延遲 + 更多成本。

  • 該加大思考預算的時機： 複雜演算法設計、多步邏輯推理、跨檔案的棘手 Bug、需要權衡多種方案的架構決策。這些任務「想得越久，越值得」。
  • 該關小思考預算的時機： 任務簡單明確、答案幾乎是機械式的（如格式轉換、命名、套樣板）。此時加大預算只會「拖慢速度、白燒 token」。
🧭 心法——依任務難度調整思考預算： 把思考預算想成「車子的油門」。直線好路（簡單任務）輕踩省油；遇到陡坡彎道（複雜推理）才大力加油。預設給中等預算，只在「卡關」或「賭注高」時才往上調。

📋 Plan Mode (規劃模式)

送出 prompt 前的第三個抉擇是「用哪個模式」。Plan Mode 的核心精神是——先規劃、再動手。

  • 不要讓 AI 直接噴 Code： 先請它產出一份實作計畫與檔案異動清單，而非劈頭就改一堆檔案。
  • 人工 Approve 是關卡： 在 AI 動工前，由你（Human-In-The-Loop）審閱藍圖；不合理就請它「更新 Plan」，滿意了才回覆 Approve。
  • 分步執行： 計畫通過後一次只執行一個 Step，讓每一步都可控、可驗證。

這是大型任務避免「義大利麵程式碼」的關鍵模式選擇。

prompt.md
# 啟動 Plan Mode
進入 Plan Mode。不要寫 Code。
先產出 ImplementationPlan.md（含檔案異動、架構、步驟）。
我回覆 Approve 後，再分步執行。

⚠️ 常見誤區與防禦

這三個抉擇若做反了，輕則燒錢拖時，重則產出一堆難以維護的程式碼。以下是最常見的三個地雷：

🚫 誤區一：無腦用最貴旗艦

不分任務難度，永遠掛著最強模型。

  • 簡單任務付出旗艦級成本與延遲
  • 批次任務的帳單迅速失控

防禦： 預設用均衡模型，只在卡關時才升級。

🚫 誤區二：簡單任務開滿思考預算

機械式的小任務也把思考預算拉到最大。

  • 回應變慢，互動體驗變差
  • 白白燒掉大量 token

防禦： 依任務難度調整，簡單任務果斷關小。

🚫 誤區三：大任務略過 Plan Mode

大型改動不先規劃，直接讓 AI 動工。

  • 產出失控的「義大利麵程式碼」
  • 異動範圍不明，難以審查與回滾

防禦： 先 Plan、人工 Approve、再分步執行。

📝 課末作業：同任務、不同大腦

紙上談兵不如親手感受。這份作業要你親眼看見「選對大腦」帶來的差異。

  • 挑一個任務： 選一個有點難度但範圍明確的任務（例如：實作一個含例外處理的小型功能，或修一個跨函式的 Bug）。
  • 用至少兩個不同等級的模型執行： 例如「輕量模型」對上「旗艦模型」，餵入完全相同的 User Prompt。
  • 記錄三個維度： 比較產出的品質（正確性、架構、可維護性）、速度（回應延遲）與成本（token 用量）。
  • 寫下你的觀察： 哪個任務「值得」用旗艦？哪個其實輕量模型就綽綽有餘？把這份直覺內化成你的選型心法。
🎯 作業目標： 讓「選模型 / 給預算 / 挑模式」從知識，變成你按下送出前的肌肉記憶。
