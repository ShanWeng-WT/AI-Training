確保 AI 不退化

Harness Engineering 與評測基準

什麼是 AI 退化災難？

當 Prompt 或模型微調後，AI 在特定功能上的表現可能突然倒退，甚至忘記 Coding 規範。我們必須量化並監控其輸出品質。

「昨天跑得很完美，今天 AI 卻突然變笨或寫出錯誤 API。」

🚨 常見的 AI 退化症狀：

  • 語法崩壞： 輸出不完整的程式碼、遺漏括號或符號，呼叫不存在的幽靈 API。
  • 幻覺發作： 憑空捏造完全無關的類別與方法。
  • 金魚腦效應： 遺忘或忽略先前已規定好的 Coding Guidelines。
  • 效能地雷： 寫出不符合優化規範的代碼（如 Unity 中在 Update 內頻繁使用 GetComponent 或產生 GC 災難）。

Harness Engineering 核心概念

如同用單元測試保護程式碼，我們也需要建立 AI 的評測基準線（Ground Truth），以客觀指標評估 Prompt 的修改成效。

「用保護程式碼的態度保護 AI：建立 AI 的單元測試。」

🎯 Ground Truth (黃金標準) 的三大支柱：

  • 標準輸入 (Input)： 標準化且經過設計的測試 Prompt，確保測試起跑點一致。
  • 預期輸出 (Expected Output)： 由資深工程師撰寫的正確示範代碼，作為對比標的。
  • 評分標準 (Rubrics)： 明確的及格條件、加分項目與應避免的反模式 (Anti-patterns)。

兩大評測方法論

結合「確定性評估」的客觀精準與「LLM 裁判」的靈活風格審查，建立全面的防護網。

🛠️ 評估模式雙軌制：

  • 確定性評估 (Deterministic Evaluation)：
    • 方法： 檢查產出是否能順利編譯、是否包含必要關鍵字 (如 AddForce)、是否包含反模式關鍵字 (如 GameObject.Find)。
    • 優點： 100% 客觀、速度極快、成本為零。
  • LLM 裁判 (LLM-as-a-Judge)：
    • 方法： 使用 GPT-4o 扮演 Tech Lead 進行 Code Review，評估架構合理性、命名規範與註解清晰度。
    • 優點： 能處理開放式、非黑即白的優雅度評分；缺點： 有些微隨機性。

實作流程：JSON + Script

藉由 JSON 定義測試集，並透過自動化腳本進行生成與評分，快速找出程式碼中的效能地雷與語法錯誤。

📋 評測資料集 JSON 範例：

[
  {
    "id": "unity_jump_001",
    "prompt": "撰寫 PlayerJump.cs，用 Rigidbody，1 秒 CD。",
    "required_keywords": ["Rigidbody", "AddForce"],
    "anti_patterns": ["GameObject.Find"]
  }
]

⚙️ 評分流程說明：

自動化測試腳本讀取 Prompt ➔ 呼叫 AI 生成程式碼 ➔ 執行編譯與關鍵字過濾 ➔ LLM 裁判針對剩餘風格評分 ➔ 輸出總成績單。

常見陷阱與防禦

防範評測系統自身的盲點，包括過度擬合、裁判主觀偏見，以及確保 Unity 測試在正確的執行緒上運作。

🛡️ 評測系統防護策略：

  • 防範過度擬合 (Overfitting)： 測試集必須涵蓋多元面向（如演算法、UI、資料處理），避免 AI 僅對特定 Prompt 優化。
  • 消除裁判偏見： LLM 裁判有時會偏愛較長程式碼。需在 Prompt 明確限制「達標即可，短非原罪」，著重功能與效能而非字數。
  • Unity 執行緒限制： 部分 Unity API 僅能在 Main Thread 執行，寫測試 Runner 時需注意執行緒調度，避免不必要的報錯。
💡 若預算有限，先落實「確定性評估」把關編譯與關鍵字，即可涵蓋 80% 的程式碼保護力。
