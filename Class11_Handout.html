確保 AI 不退化

Harness Engineering 與評測基準

一、什麼是 AI 退化災難？

昨天完美，今天報錯。
  • 語法崩壞： 遺漏符號，呼叫幽靈 API。
  • 幻覺發作： 憑空捏造類別與方法。
  • 金魚腦： 忘記 Coding Guidelines。
  • 效能地雷： 寫出 GC 災難代碼。

二、Harness Engineering 核心概念

用保護程式碼的態度保護 AI：建立 AI 的單元測試。
  • Ground Truth： 客觀的黃金基準。
  • Input： 標準化測試 Prompt。
  • Expected Output： 資深工程師的正確示範。
  • Rubrics： 明確及格條件與反模式。

三、兩大評測方法論

  • 確定性評估：
    • 非黑即白：編譯與單元測試。
    • 優點：100% 客觀、極速。
  • LLM 裁判：
    • 讓 GPT-4o 審查架構與風格。
    • 優點：宛如資深工程師 Code Review。
    • 缺點：具些微隨機性。

四、實作流程：JSON + Script

先抓關鍵字與反模式，再請 LLM 裁判綜合評分。
[
  {
    "id": "unity_jump_001",
    "prompt": "撰寫 PlayerJump.cs，用 Rigidbody，1 秒 CD。",
    "required_keywords": ["Rigidbody", "AddForce"],
    "anti_patterns": ["GameObject.Find"]
  }
]

五、常見陷阱與防禦

  • 過度擬合： 考題單一 👉 涵蓋演算法與 UI 多面向。
  • 裁判偏見： 偏愛長代碼 👉 規則明訂「達標即可，短非原罪」。
  • 執行緒限制： Unity API 報錯 👉 確保測試僅在 Main Thread 執行。
