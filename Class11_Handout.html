確保 AI 不退化

Harness Engineering 與評測基準

一、什麼是 AI 退化災難？

現象： 昨天還能寫出完美程式碼的 Prompt，今天卻產出語法錯誤或邏輯瑕疵。

  • 語法崩壞： 遺漏分號、括號，呼叫不存在的 API。
  • AI 幻覺 (Hallucination)： 捏造了引擎中根本沒有的類別與方法。
  • 上下文遺忘： 在長對話中忘記了團隊的 Coding Guidelines。
  • 效能地雷： 寫出在 `Update()` 頻繁 `GetComponent` 的 GC 災難代碼。

二、Harness Engineering 核心概念

「用保護程式碼的態度保護 AI：建立 AI 的單元測試。」

  • Ground Truth (黃金標準)： 評測的客觀基準。
  • 輸入 (Input)： 標準化的 Prompt。
  • 預期輸出 (Expected Output)： 資深工程師寫好的正確示範。
  • 評分標準 (Rubrics)： 明確定義及格條件 (例如必備關鍵字、不可使用的反模式)。

三、兩大評測方法論

  • 確定性評估 (Deterministic Evaluation)：
    • 非黑即白：編譯測試 (dotnet build)、單元測試。
    • 優點：100% 客觀、快速。
  • LLM 作為裁判 (LLM-as-a-Judge)：
    • 使用強大模型 (如 GPT-4o) 審查架構、可讀性與註解風格。
    • 優點：接近資深工程師的 Code Review。
    • 缺點：具些微隨機性。

四、實作流程概覽 (JSON + Script)

透過 JSON 定義資料集，使用腳本進行自動化計分。

[
  {
    "id": "unity_jump_001",
    "prompt": "撰寫 PlayerJump.cs，使用 Rigidbody，並有 1 秒 CD。",
    "required_keywords": ["Rigidbody", "AddForce", "Space"],
    "anti_patterns": ["GameObject.Find"]
  }
]

執行邏輯： 先檢查關鍵字與反模式 (扣分制)，接著呼叫 LLM 裁判針對 Rubrics 進行綜合打分與給予改進建議。

五、常見陷阱與防禦

  • 過度擬合 (Overfitting)： 資料集過於單一。防禦： 涵蓋演算法、架構、UI 多種面向。
  • LLM 裁判偏見 (Judge Bias)： 偏好長篇大論。防禦： 在 Prompt 中明令「只要邏輯正確、效能達標即可，不因代碼短而扣分」。
  • Unity 執行緒限制： 防禦： 確保測試腳本中的 Unity API 呼叫僅於 Main Thread 執行。
