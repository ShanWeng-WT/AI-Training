從對話到行動

Agent 的本質與運作機制總結

🎯 課程目標

我們把 ChatGPT 當作聊天機器人，但我們給開發者的工具是 Agent。兩者最大的差別在於「行動力」。

  • 看透 Agent 的底層本質： 釐清 Chatbot（只會預測文字）與 Agent（擁有自主行動力）的核心差異。
  • 解構 ReAct 推理迴圈： 深入學習經典的「感知 - 思考 - 行動 (Reasoning & Acting)」運作邏輯。
  • 階段性技術總結： 統整前 6 堂課的 Agent 開發心法，為邁向企業級專案大腦做準備。

📖 什麼是 Agent？

Chatbot 只能說話，Agent 能夠行動。要讓一個語言模型能替人類工作，必須為它裝上「手」和「腳」。

  • Chatbot (聊天機器人)： 接收輸入，靜態預測並輸出下一段文字，無法感知外部環境，也無法產生實際的副作用。
  • Agent (代理人)： 擁有明確的任務目標，能主動呼叫外部工具 (Tools)，並依據執行結果進行自我反思與修正。
  • Agent 的三大支柱：
    • 推理大腦 (Reasoner)： LLM 本身的推理與步驟規劃能力。
    • 外部工具 (Tools)： 讓 Agent 具備對系統、檔案與 API 的操控權限。
    • 反思修正 (Reflection)： 讀取工具報錯並動態變更開發路線。

💡 頓悟時刻

Agent 不是什麼高深莫測的魔法，它本質上就是一個掛了 LLM 推理大腦的 while 迴圈！

底層運作邏輯虛擬碼： 
while (!task_completed) {
    let thought = think(goal, history);
    let action = select_tool(thought);
    let observation = execute(action);
    update_history(thought, action, observation);
    task_completed = evaluate(goal, history);
}
這解釋了為什麼它有時候會「想很久」，因為它正在背後瘋狂呼叫工具、讀檔、報錯、自我修正。

🛠️ 核心機制：ReAct 框架

Reasoning and Acting (ReAct) 是 Agent 最著名的運行框架。學會閱讀 Agent 的隱藏 Trace Log，是開發者最重要的除錯基本功。

  • Thought (思考)： 分析當前狀況與目標差距，推理下一步該做什麼。
  • Action (行動)： 決定調用的工具 (Tool) 並傳入所需的引數 (Parameters)。
  • Observation (觀察)： 工具執行後的實質反饋（如：檔案內容、編譯器報錯、命令執行結果）。
開發者啟示： 要讓 Agent 的表現穩定且強大，除了寫好系統 Prompt，「提供功能單一、防呆完善、回傳格式化錯誤訊息的 Tools」更是成功的關鍵。

⚠️ 為什麼 Agent 會卡死？

當 Agent 陷入無效的重複思考或無限的 retry 迴圈時，我們必須介入並優化我們的指令與工具設計。

  • 工具不穩定與無防呆：
    • 工具噴出未處理的 Exception 或格式混亂的錯誤訊息，AI 無法解讀，只會以同樣的方式反覆嘗試，導致陷入死鎖。
  • 目標模糊 (Goal Ambiguity)：
    • 任務目標不夠具體（例如「幫我優化專案程式碼」），Agent 不知道什麼樣的狀態才算「完成」，因而無休止地運行下去。
  • 對策： 在提示詞中寫明任務完成的驗證條件（如「當執行 dotnet build 通過且無編譯錯誤時即代表任務完成」），並對工具加上明確的 Exception Handling。
