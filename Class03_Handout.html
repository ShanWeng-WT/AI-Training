揭開 LLM 的黑盒子：它能做什麼，不能做什麼？

企業內部 AI 輔助開發教育訓練

🎯 課程目標 Session Goals

剖析生成式大語言模型的底層極限，防範並收斂「幻覺」以獲得穩定的工程產出。

本堂課將帶領大家看清 LLM 的本質，轉變對 AI 輸出的預期與使用思維：

  • 理解底層： 揭開 LLM 的運作機制，了解它並非真的懂語意，而是依賴統計機率。
  • 認識極限： 深入理解「幻覺」發生的數學根本原因，避免盲目信任 AI 產出。
  • 建立信任： 學會以「讀 diff、Build/Test、Git 審查」等手動驗證手段，把機率性的產出收斂為可信任的結果。

📖 LLM 核心概念

LLM 本質上是文字接龍引擎。它沒有「真偽」的觀念，只有「合理」的機率分佈。

🤖 超級文字接龍

LLM 本質上不具備邏輯推理或程式碼編譯能力，它只是基於前文預測下一個最可能出現的 Token。

⚙️ 以 Token 為處理單位

AI 理解代碼的最小單位不是單字或字母，而是如 =>、public 等 Token，這決定了其語義匹配特徵。

💡 沒有真實，只有合理

當模型缺乏特定上下文時，會硬生生捏造看似語意通順但不存在的 API。這就是幻覺 (Hallucination) 的物理本質。

🛠️ 實戰： 對抗幻覺

以 Unity 尋找最近敵人並扣血為例，展示缺乏約束與加入約束的產出差異。

情境： 尋找最近敵人並觸發傷害。

  • ❌ 缺乏約束： AI 隨機接龍，捏造出不存在的 Physics.FindClosestComponent API。
  • ✔️ 加入約束： 強制約束「必須使用 OverlapSphere 並透過 IDamageable 介面觸發傷害」。
example.cs
// 透過 Constraints 限縮機率後，AI 穩定產生正確代碼
Collider[] hits = Physics.OverlapSphere(pos, radius);
foreach (var hit in hits) {
    if (hit.TryGetComponent<IDamageable>(out var target)) {
        target.TakeDamage(damageAmount);
    }
}

🔍 手動驗證與 審查手段

因為 LLM 是機率性的、必然會產生幻覺，所以「驗證與審查」不是不信任 AI，而是建立信任的工程手段。AI 說「好了」不算數，要由人類與工具給出確定性的真理。

📄 讀 Diff

審查 AI 的每一次變更，透過 git diff 或 IDE 的 diff view，只聚焦「改了什麼」而非重讀全部檔案。
✔️ 習慣： 培養快速審 diff 的肌肉記憶，讓每行改動都經過人眼確認。

⚙️ Build / Test 驗證

把編譯器與單元測試當作確定性的真理來源，這是機率無法欺騙的客觀事實。
✔️ 標準： AI 說「好了」不算數，Build 過 + 測試綠燈才算真的完成。

🌿 用 Git 審 AI Commit

讓 AI 的每個邏輯單元獨立 commit，方便逐項 review、回溯與 revert。
✔️ 安全網： 把 Git 當成與 AI 協作的護欄，出錯時隨時能精準退回乾淨狀態。

⚠️ 常見誤區與 防禦機制

生成式模型的創造力與幻覺是一體兩面的。防禦性開發思維是必不可少的護欄。

🤖 依賴 AI 進行複雜計算

❌ AI 不擅長運算或推演複雜狀態機。
✔️ 防禦： 讓 AI 搭建架構或寫單元測試，核心計算交由人工補足或實體編譯驗證。

⚙️ 輕信預設的套件與 API 版本

❌
✔️ 防禦： 必須在 context 中明文指定框架版本與依賴限制（例如：C# 12 / .NET 8）。

💡 塞入過多不相干的雜訊

❌ 導致注意力分散。
✔️ 防禦： 保持高訊噪比，只餵給 AI 必要且精確的專案結構與文件。

❓ 常見問答 Q&A

解惑開發團隊對大語言模型本質與限制的疑惑。

❌ Q1: AI 真的理解它寫的程式碼嗎？未來會完全沒有幻覺嗎？


A1: 不，它不理解。它基於強大的「模式匹配」捕捉關聯。而幻覺與創造力是一體兩面的，即使到 GPT-5，幻覺也無法完全根除，因此防禦性提示詞與人工審查依然至關重要。

🎯 Q2: 既然要逐行讀 diff、跑 Build/Test，AI 還有幫我省到時間嗎？


A2: 有，而且是兩種不同的工作。AI 把你從「從零撰寫」變成「審查與驗證」，這在認知負擔上輕得多。配合小步 commit + 綠燈測試，審查成本被攤平，整體仍大幅加速，且品質更可控。
