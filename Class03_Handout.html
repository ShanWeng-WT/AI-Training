揭開 LLM 的黑盒子：它能做什麼，不能做什麼？

企業內部 AI 輔助開發教育訓練

🎯 課程目標 Session Goals

「你不會被AI取代，你只會被會用AI的人取代。」

本堂課將帶領大家看清 LLM 的本質，轉變對 AI 輸出的預期與使用思維：

  • 理解底層： 揭開 LLM 的運作機制，了解它並非真正會思考，而是依賴統計推測與機率。
  • 認識極限： 深入理解「幻覺」發生的物理本質以及 Token 等特性的影響，避免盲目信任 AI。
  • 建立信任： 學會以「讀 diff、Build/Test、Git 審查」等工程手段，把機率性的產出收斂為可信結果。

📖 AI 如何思考？ 核心概念

AI 並非真正會思考，他會的是統計及推測。當資料量極大時，便產生了「湧現現象」。

🤖 超級文字接龍

LLM 依賴統計與推測，基於前文預測下一個最可能出現的詞。它沒有「真偽」的觀念，只有「合理」的機率分佈。

🌟 湧現現象

許多小實體交互作用後產生了大實體，展現原本不具有的特性。當學習資料量大到某個量級，會產生如「頓悟」般的表現。

💬 與機器對話

即便看似像人，本質上還是演算法與海量資料。跟 AI 溝通就像寫程式一樣，有特定的方法與限制需要掌握。

🔑 影響輸出的 關鍵特性

理解影響 LLM 輸出的核心機制，有助於我們更好地操控 AI。

🧩 Token 機制

AI 處理文字的最小單位不是單字而是 Token。這導致它算數不太行、無法拆字，且Prompt 的語言選擇及錯字都會大幅影響預測機率與成本。

🎲 隨機性

AI 透過機率分布選擇最可能的下一個 Token，因此「正確答案」不保證機率最高，且每次產出可能都不同（受 Temperature 參數影響）。

⏳ 無狀態 (Stateless)

AI 沒有記憶，每次 prompt 都是獨立的。需要依賴 System Prompt 定義角色，並提供完整的 Context (上下文) 來維持對話脈絡。

⚠️ 幻覺 (Hallucination) 與對策

AI 可能一本正經地胡說八道，因為它是依機率生成文字，而非真正理解。我們必須採取防禦性開發思維。

❌ 幻覺的表現


常見狀況包括：答非所問、捏造不存在的 API 或事實、提供錯誤但聽起來合理的資訊，甚至在你提供錯誤資訊時依然順著你說對。這多源於訓練資料品質、脈絡缺乏或 Prompt 引導錯誤。

🛡️ 防禦對策


✔️ 不盲信：要求 AI 引用來源，不要依賴 AI 進行複雜計算。
✔️ 調整參數：適當降低 Temperature。
✔️ 提升品質：提供充足Context、明確指定框架版本 (如 C# 12 / .NET 8)，並保持高訊噪比，減少無關雜訊。

🛠️ 實戰： 對抗幻覺

以 Unity 尋找最近敵人並扣血為例，展示缺乏約束與加入約束的產出差異。

情境： 尋找最近敵人並觸發傷害。

  • ❌ 缺乏約束： 當遇到模糊問題「物件沒顯示」或指令不清時，AI 可能亂猜，甚至捏造出不存在的 Physics.FindClosestComponent API。
  • ✔️ 加入約束： 強制約束「必須使用 OverlapSphere 並透過 IDamageable 介面觸發傷害」。
example.cs
// 透過 Constraints 限縮機率後，AI 穩定產生正確代碼
Collider[] hits = Physics.OverlapSphere(pos, radius);
foreach (var hit in hits) {
    if (hit.TryGetComponent<IDamageable>(out var target)) {
        target.TakeDamage(damageAmount);
    }
}

🔍 手動驗證與 審查手段

AI 可靠度夠嗎？其實人也一直在犯錯，AI 只是工具。把節省的時間用於驗證與打磨品質，AI 說「好了」不算數，要由人類與工具給出確定性。

📄 讀 Diff

審查 AI 的每一次變更，透過 git diff 或 IDE 視窗，只聚焦「改了什麼」。
✔️ 習慣： 培養快速審 diff 的肌肉記憶，讓每行改動都經過人眼確認。

⚙️ Build / Test 驗證

把編譯器與單元測試當作確定性的真理來源，這是機率無法欺騙的客觀事實。
✔️ 標準： AI 說「好了」不算數，Build 過 + 測試綠燈才算真的完成。

🌿 用 Git 審 AI Commit

讓 AI 的每個邏輯單元獨立 commit，方便逐項 review、回溯與 revert。
✔️ 安全網： 把 Git 當成與 AI 協作的護欄，出錯時隨時能精準退回乾淨狀態。

❓ 常見問答 Q&A

解惑開發團隊對大語言模型本質與限制的疑惑。

❌ Q1: AI 真的理解它寫的程式碼嗎？未來會完全沒有幻覺嗎？


A1: 不，它不理解。它基於強大的「模式匹配」捕捉關聯。幻覺與創造力是一體兩面的，幻覺無法完全根除，因此防禦性提示詞與人工審查依然至關重要。

🎯 Q2: 既然要逐行讀 diff、跑 Build/Test，AI 還有幫我省到時間嗎？


A2: 有的。AI 把你從「從零撰寫」變成「審查與驗證」，認知負擔輕得多。配合小步 commit + 綠燈測試，審查成本被攤平，整體仍大幅加速，且品質更可控。
