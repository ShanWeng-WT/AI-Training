第 12 堂：自動化測試你的 AI

實作 Evaluation Harness 與 CI/CD 整合

一、為何需要自動化測試 AI？

「如果沒有量化準確率，AI 永遠只是玩具，無法成為工業級工具。」

  • 防範回歸 (Regression)： 修改 Prompt 或系統設定後，確保舊任務不崩壞。
  • 量化指標： 提供明確的數據 (如 90% 成功率)，判斷 AI Skill 是否可上線。
  • 建立信任： 透過透明報告降低團隊對 AI 黑盒子的不信任。

二、Evaluation Harness 的三大要素

批次考試系統，用來量化評估 Agent 表現。

  • Dataset (資料集)： 預期的輸入任務與 Ground Truth。
  • Runner (執行器)： 批次餵給 Agent 任務並取得輸出。
  • Evaluator (評分器)： 比對輸出，如精確匹配 (Exact Match)、編譯測試。

三、CI/CD 與 Git Flow 整合

「Agent 設定檔如同程式碼，需嚴格版控與審查。」

  • 修改流程： 修改 Agent 提示詞 ➔ 發 PR ➔ 觸發 CI 跑 Evaluation Harness。
  • 自動把關： 若準確率低於門檻 (例如 90%)，CI 回傳 Error Code 阻擋合併，並自動留言警示。
  • 實務應用： 平時只跑核心 Smoke Test，發布版本前跑 Full Evaluation。
# GitHub Actions 範例
jobs:
  evaluate:
    steps:
      - run: dotnet run # 執行 Harness
      - name: Comment on PR
        if: failure()
        run: echo "❌ AI 準確率低於標準，阻擋合併。"

四、企業級 AI 開發四大守則

  1. AI 是編譯器，不是魔術師： 產出物必須受 Lint 與單元測試檢驗。
  2. No CI, No AI： 沒有極高覆蓋率的自動化測試，就不要把 AI 產出大量放進專案。
  3. Prompt 版控： 所有 System Prompt 與 Skill 檔必須進 Git 版本控制。
  4. 沙盒執行 (Sandbox Only)： AI 產出的指令在未經人類同意前，只在 Sandbox 內執行。

五、常見陷阱與防禦機制

  • 過度優化 (Overfitting)： 為了測試 100 分，Prompt 寫死參數。防禦： 建立盲測資料集。
  • LLM 裁判幻覺： LLM 誤判給滿分。防禦： 優先依賴「編譯器、語法樹」等確定性測試。LLM 只能做輔助審閱。
