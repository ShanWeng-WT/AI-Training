從對話到行動

Agent 的本質、運作機制與日常工作環境

📖 什麼是 Agent？

LLM 本身只是大腦，無法做任何事。Agent 就是給 LLM 賦予身體，在思考能力之上再加入感知與行動能力。

🤔 想一想，下列哪些算是 Agent 呢？

  • 案例 1：給AI一篇文章，叫他總結文章重點。
  • 案例 2：在IDE裡使用代理人模式讓他直接讀寫程式碼。
  • 案例 3：AI實作完自己執行Build與Test，自己發現錯誤並修正
  • 案例 4：給予目標後，能自主規劃、建專案、寫程式碼、測試，不斷循環直到目標達成。
  • Chatbot (聊天機器人)： 接收輸入後，LLM預測並輸出下一段文字；無法感知也無法影響外部環境。
  • Agent (代理人)： 擁有明確的任務目標，能主動呼叫外部工具 (Tools)，並依據執行結果進行自我反思與修正。

Agent 的三大支柱

🧠 推理大腦 (Reasoner)

LLM 本身的推理與步驟規劃能力。

🛠️ 外部工具 (Tools)

讓 Agent 具備對系統、檔案與 API 的操控權限。

💡 反思修正 (Reflection)

讀取工具報錯並動態變更開發路線。

🛠️ Agent 運作架構：ReAct 框架

Reasoning and Acting (ReAct) 是 Agent 最著名的運行框架。學會閱讀 Agent 的隱藏 Trace Log，是開發者最重要的除錯基本功。

🤖 Agent 的能力範圍與邊界

Agent 是嵌入在一個指定環境 (Environment) 中運作的個體：它透過 ReAct 的 Observation 感知、Action 行動與環境互動。也因此，它的能力上限不只取決於模型，更取決於：

🧠 Context（可見脈絡）

包含對話歷史、System Prompt、AGENTS.md與工具回傳的Result(Observation與Action的結果)。看不到的資訊，它就無從判斷，僅能依賴訓練資料。

🛠️ Tools（可用工具）

讀檔、搜尋、修改、執行測試、瀏覽網頁等工具。沒有對應工具，Agent就無法對環境做觀察(Observation)與行動(Action)，僅能依賴訓練資料。

🧑‍⚖️ 範圍與授權

界定Agent作用的範圍；權限決定工具是否能對環境執行Observation與Action。

🎮 範例：台北今天適合跑步嗎？

Agent_Trace_Log.txt
[User Prompt] 台北今天適合跑步嗎？

[Thought] 要回答這個問題，我需要知道今天是哪天，目前是在Windows環境，我可以使用echo %date%指令取得今天的日期。
[Action] 呼叫指令echo %date%
[Observation] 指令回覆「202-06-22」。

[Thought] 光是知道今天是哪天還不足以判斷是否適合跑步，我還需要知道台北的天氣，我可以使用天氣工具取得資訊。
[Action] 呼叫天氣工具，帶入參數「Taipei」,「2026-06-22」。
[Observation] 工具回覆「30 度，多雲，有午後雷陣雨」。+

[Thought] 今天可能會有雷陣雨，所以不適合戶外跑步，已確認可以回答使用者的問題了。
[Final Answer] 今天不建議在台北跑步，因為可能會下雨。

🤔 延伸思考

  • 如果 User Prompt 改成「今天適合跑步嗎？」會變成怎麼樣？
  • 如果 Agent 沒有天氣工具可以用會怎麼樣？
  • Agent最終的回答是不是真的能夠滿足使用者的需求？說不定他是想在清晨跑步；又或者今天不會下雨，但是他要去的田徑場今天不開放

⚖️ Agent vs 純 LLM：同一個問題，兩種做法

💬 純 LLM

僅憑訓練資料回答，受限於Knowledge cutoff，不可能知道「今天」台北的即時天氣，容易產生幻覺；優點是不需呼叫外部工具，回應快、成本低。

🤖 Agent (ReAct)

能呼叫天氣工具取得即時觀測值，可信度更高；代價是需要額外的工具整合與授權，且內部多輪的循環也使得回應時間與Token成本大幅增加。

沒有絕對的優劣之分，只有適用情境的不同

⚠️ 為什麼 Agent 會卡死？

當 Agent 陷入無效的重複思考或無限的 retry 迴圈時，我們必須介入並優化我們的指令與工具設計。

  • 工具不穩定與無防呆：
    • 工具噴出未處理的 Exception 或格式混亂的錯誤訊息，AI 無法解讀，只會以同樣的方式反覆嘗試，導致陷入死鎖。
  • 目標模糊 (Goal Ambiguity)：
    • 任務目標不夠具體（例如「幫我優化專案程式碼」），Agent 不知道什麼樣的狀態才算「完成」，因而無休止地運行下去。
  • 對策： 在提示詞中寫明任務完成的驗證條件（如「當執行 dotnet build 通過且無編譯錯誤時即代表任務完成」），並對工具加上明確的 Exception Handling。
