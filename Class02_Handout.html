Agent 的本質、運作機制與日常工作環境

📖 什麼是 Agent？

LLM 本身只是大腦，無法做任何事。
Agent 就是給 LLM 賦予身體，在思考能力之上再加入感知與行動能力。

🔍 LLM、Chatbot、Agent 有何不同？

🧠 LLM

The Brain — 核心引擎
  • 理解並生成自然語言
  • 憑訓練資料推理
  • 無法主動感知或行動
  • 例：GPT-5、Claude、Gemini

💬 Chatbot

The Interface — 對話介面
  • 在對話窗口中與使用者互動
  • LLM 驅動，以文字問答為主
  • 單輪或多輪對話，不主動執行
  • 例：ChatGPT 網頁聊天介面

🤖 Agent

The Doer — 行動執行者
  • 以 LLM 為大腦，外掛工具為手腳
  • 可讀寫檔案、呼叫 API、執行程式
  • 透過 ReAct 迴圈反覆觀察與修正
  • 例：Claude Code、Codex、Antigravity
LLM（核心） → 驅動 → Chatbot（包裝對話） → 擴充工具與行動 → Agent（能感知並執行）

🤔 想一想，下列哪些算是 Agent 呢？

  • 案例 1：給AI一篇文章，叫他總結重點。
  • 案例 2：在IDE裡使用代理人模式讀寫程式碼。
  • 案例 3：AI實作完自己執行Test，發現錯誤並修正
  • 案例 4：給予目標後，能自主規劃、建專案、寫程式碼、測試，不斷循環直到目標達成。
會不會分辨不是重點
釐清 自己的需求 該用什麼 手段達成 才是關鍵

🛠️ 什麼是 Tool Use (Function Calling)？

🌟 核心概念

讓 LLM 能根據使用者需求，呼叫額外的函式或 API，突破只能文字生成的限制。


通常除了Harness會內建通用工具以外(如:檔案讀/寫/搜尋，Web Fetch等)，其他工具皆需要使用者透過Skill, Plugin, MCP 或自行製作以提供給Agent使用

🎮 互動流程 (遊戲開發範例)

Agent 無法直接「看見」遊戲場景，必須透過工具與環境一來一往互動：

User: 幫我在玩家角色面前新增一個敵人的物件
Agent: 決定呼叫工具 GetPosition("Player")
Tool : 執行後回傳結果 Position(10, 0, 5)
Agent: 以回傳結果判斷新生成敵人的位置，決定呼叫工具SpawnObject("Enemy")
Tool : 執行後回傳結果 SpawnResult(Success, "Enemy")
Agent : 確認工具執行成功，回報使用者結果


延伸思考: 叫Agent執行工具，和直接寫一個Editor Script來執行有什麼差?

🔍 資訊檢索型

取得最新或特定領域資訊。

  • 網頁搜尋 (Google Search)
  • 資料庫查詢 (SQL)
  • 取得天氣、股價 API

⚙️ 系統操作型

改變環境狀態或執行運算。

  • 讀寫本機檔案 (File System)
  • 執行終端機指令 (Terminal)
  • 發送 Email 或 Slack 訊息

🧠 邏輯運算與專業工具

彌補 LLM 在計算上的不足或特定功能。

  • Python 執行器 (用於資料分析或繪圖)
  • 計算機工具 (Calculator)
  • 編譯與測試工具 (dotnet build, npm)

🛠️ Agent 運作架構：ReAct 框架

Reasoning and Acting (ReAct) 是 Agent 最著名的運行框架。
Agent 透過不斷循環以下三個步驟，讓大腦指揮手腳來解決複雜問題：

🧠 Thought (思考)

分析當前狀況與目標差距，推理下一步該做什麼。

⚙️ Action (行動)

決定調用哪一個外部工具 (Tool) 並傳入對應的參數。

💡 Observation (觀察)

工具執行後傳回的實質反饋，作為下一輪思考的輸入。

🎮 Trace Log 範例：台北今天適合跑步嗎？

Agent_Trace_Log.txt
[User Prompt] 台北今天適合跑步嗎？

[Thought] 要回答這個問題，我需要知道今天是哪天，目前是在Windows環境，我可以使用echo %date%指令取得今天的日期。
[Action] 呼叫指令echo %date%
[Observation] 指令回覆「2026-06-22」。

[Thought] 光是知道今天是哪天還不足以判斷是否適合跑步，我還需要知道台北的天氣，我可以使用天氣工具取得資訊。
[Action] 呼叫天氣工具，帶入參數「Taipei」,「2026-06-22」。
[Observation] 工具回覆「30 度，多雲，有午後雷陣雨」。

[Thought] 今天可能會有雷陣雨，所以不適合戶外跑步，已確認可以回答使用者的問題了。
[Final Answer] 今天不建議在台北跑步，因為可能會下雨。

🤔 延伸思考

  • 如果 User Prompt 移除「台北」會變成怎麼樣？
  • 如果 Agent 沒有天氣工具可以用會怎麼樣？
  • Agent 最終的回答是不是真的能夠滿足使用者的需求？說不定他是想在清晨跑步；又或者今天不會下雨，但是他要去的田徑場今天不開放?

🤖 Agent 的能力範圍與邊界

Agent 是嵌入在一個指定環境 (Environment) 中運作的個體：它透過 ReAct 的 Observation 感知、Action 行動與環境互動。也因此，它的能力上限不只取決於模型，更取決於三個環境要素：

  • 🧠 Context（可見脈絡）
    包含對話歷史、System Prompt、專案架構文件與工具回傳的結果。看不到的資訊，它就無從判斷，僅能瞎猜或依賴訓練資料。
  • 🛠️ Tools（可用工具）
    讀檔、搜尋、修改、執行測試、瀏覽網頁等能力。沒有對應工具，Agent 就無法觀察或行動，空有聰明大腦也無能為力。
  • 🧑‍⚖️ Scope & Auth（範圍與授權）
    界定 Agent 作用的環境邊界。即使有工具，權限的開關也決定了它是否能真正對環境造成改變。

⚠️ 使用 Agent 的常見問題

🧑‍💻 開發者
我的Agent一直在無限 retry ，任務都做不完？
🤖 AI
如果你不想燒光Token或讓Context髒掉的話，就趕快讓他停下！
這通常是因為 Agent 無法從環境中得到有效的 Observation！常見的雷區有：

1. 目標模糊：
任務如果不夠具體（例如只說「幫我優化程式碼」），Agent 就不知道什麼狀態才算「完成」。
💡 解法：說清楚要優化什麼，或在指令寫明驗證條件（例如「當 dotnet build 通過即代表完成」）。

2. 缺乏脈絡、資訊不對稱 (雞同鴨講)：
開發者給錯方向，或Agent所在的環境或工具限制了他的能力邊界，導致他找不到有效資訊而整個歪樓，注意力分散。
💡 解法：開發者需要確保Agent能取得必要脈絡 (Context Engineering)，引導他正確方向。

3. 工具問題：
沒有合適的工具，或者工具噴出無用的Log、模糊的Error Message，甚至是沒有回應，會讓Agent一直做無用的嘗試。
💡 解法：開發 Tool 時，務必設計好 Error Handling 與易懂的錯誤訊息。

🧑‍💻 開發者
我怎麼知道我的 Agent 有哪些工具可用？如果沒有適合的工具怎麼辦？
🤖 AI
在你使用的Harness的介面或官方文件中尋找相關的說明，或直接在對話中詢問 Agent：「請列出你目前支援的所有工具及功能說明」。

如果發現沒有合適的工具：
💡 1. 檢查是否因為權限不足 (Scope & Auth) 而導致工具未開放。
💡 2. 自己實作對應的 Tool 介面，擴充 Agent 的能力！
🧑‍💻 開發者
如果我的 Agent 執行了危險操作，把專案改壞了怎麼辦？
🤖 AI
這就是為什麼我們需要護欄 (Guardrails)！我們不該讓 Agent 在無保護的環境下裸奔。

💡 防線 1 - 版本控制 (VCS)：讓 Agent 在獨立分支工作，就算改壞了也能還原。
💡 防線 2 - Human-in-the-loop (HITL)：對於高風險行為，強制 Agent 發出請求，獲得人類許可 (Approve) 才能執行。
💡 防線 3 - 使用沙盒 (Sandbox)：讓 Agent 在分離環境中執行，直接讓他摸不著也看不見。

⚖️ Agent vs 純 LLM：同一個問題，兩種做法

💬 純 LLM

僅憑訓練資料回答，受限於Knowledge cutoff，不可能知道「今天」的即時天氣，容易產生幻覺；優點是不需呼叫外部工具，回應快、成本低。

🤖 Agent (ReAct)

能呼叫外部工具取得即時觀測值，可信度更高且能實際執行任務；代價是需要額外的工具整合與授權，且內部多輪的循環也使得回應時間與 Token 成本大幅增加。

🔌 MCP (Model Context Protocol) 基礎介紹

🤔 為什麼要使用 MCP ？

因為許多軟體的核心 API 都不對外開放或無法輕易呼叫(如Unity)。
因此我們必須在目標環境中安置一個「接口」提供給 Agent，也就是 MCP Server。

🤔 開發者疑問：我們大可在 Unity 裡面寫一個簡單的 REST API（例如開在 localhost:8080），然後在 Agent 那端寫一個 Tool給Agent發GET請求就好了。
自己刻 API 及 Tool 就能動，那遵守 MCP 協議的用意在哪？

🔌自動發現與註冊

Plug-and-Play

Agent 只要連上 MCP Server，會自動發送一個 ListTools 的標準指令。

伺服器回傳：「我有 GetHierarchy 和 SpawnCube 這兩個工具，參數分別是...」

Agent 瞬間學會新技能，完全不需要在 Agent 端寫任何一行對接的綁定程式碼。

🧱標準化的三大積木

Resources / Tools / Prompts
MCP 嚴格定義了 Server 只能提供三種東西：
  • Resources (資源)： 提供 Context。例如: 「當前 Unity 打開的 Scene 檔案內容」。
  • Tools (工具)： 提供 Action。例如: 「在 Unity 內創建一個物件」。
  • Prompts (提示詞)： 提供預設指令。例如: 「幫我檢查 Unity 效能的 Prompt 範本」。

🌐無縫跨環境協作

Any Client ↔ Any Server

因為大家都遵守 MCP 標準，同一個 Agent (例如 Claude Desktop) 可以「同時」插上 Gmail 和 Unity 的 MCP Server。

Agent 可以讀取 Gmail 裡的企劃書 (Resource)，直接呼叫 Unity 的工具 (Tool) 來建立對應場景，中間毫無阻礙。
