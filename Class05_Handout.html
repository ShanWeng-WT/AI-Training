分工協作的藝術

Subagents 在複雜任務中的應用

🎯 課程目標

AI 就跟人一樣，一次塞太多工作會當機。解決方案就是：Subagents 分工協作。

  • 解決大任務問題： 學會將龐大、多步驟的任務拆解成可被單一 Agent 輕鬆搞定的子任務。
  • 理解 Subagent 運作： 掌握將龐大任務合理拆解，各個擊破的設計心法。
  • 實作分工協作流： 親自體驗多個 Agent 協同分工，建立系統化的協作流程。
  • 降低單一負載： 降低 Context 迷失與推理鏈中斷問題，顯著提升代碼準確率與效率。

📖 為什麼需要 Subagents？

單一 AI 模型有 Context 限制。在面對複雜的大型任務時，單打獨鬥容易迷失方向、產生幻覺。

  • 單一 AI 的上下文盲點： 當對話過長時，AI 容易忽略部分關鍵指令，或在推理鏈中途迷失。
  • 單打獨鬥的弊端： 單一 Agent 必須同時處理需求規劃、編碼與品質審查，容易顧此失彼，導致生成結果充滿幻覺。
  • 團隊作戰的優勢： 透過專精分工，將各個 Agent 的注意力限制在極窄的上下文範圍中，確保每一階段的產出質量。

💡 頓悟時刻

我們不是在寫一個更長、更複雜的 Prompt，而是在組建一間「虛擬軟體公司」。

這是一次思維的翻轉：
我們不再試圖用單次對話完成所有事，而是像管理團隊一樣：
  • PM (Router)： 拆解需求、決定工作優先順序
  • 工程師 (Coder)： 專注於生成特定模組或語言邏輯
  • QA (Reviewer)： 負責檢查程式碼，回饋修改建議
各司其職，品質才有保障！

🛠️ 常見的分工模式

未來我們面對複雜需求，第一步不再是無腦下 Prompt，而是先想「這需要幾個角色來完成？」。

  • Router (路由)： 判斷使用者意圖與需求類別，並將任務分發給最適當的 Subagent。
  • Coder (撰寫)： 專注於特定技術堆疊（例如前端 React 或後端 .NET）的代碼生成。
  • Reviewer (審查)： 靜態檢查規範、語法與潛在邏輯錯誤，提供迭代修正意見。

⚠️ 常見誤區

不要為了分工而分工。Subagent 之間的資料傳遞也是成本，我們必須在結構清晰度與系統延遲之間取得平衡。

  • 過度拆分 (Over-engineering)：
    • 小任務殺雞用牛刀（如簡單的迴圈邏輯使用三個 Subagent），反而增加延遲與系統不穩定性。
  • 溝通成本與資料遺失：
    • Subagent 之間若沒有確實傳遞上下文，容易遺失前置任務的關鍵資訊。下堂課的「Context Engineering」將教大家如何精準傳遞。
