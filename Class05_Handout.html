Context Engineering

Context Window、Lost in the Middle 與工作記憶管理

🎯 課程目標

承接上一堂「為何要選對模型」，這一堂自然帶到「為何要管理上下文」。這不是教你怎麼寫出漂亮的 Prompt，而是教你如何像工程師管理記憶體一樣，精準控制餵給 AI 的工作記憶。

  • 掌控 Context 物理限制： 了解 Context Window 與 token 限制，量化並管理 LLM 的工作記憶。
  • 對抗 Lost in the Middle： 理解中間遺忘效應，學會把關鍵規則放在模型注意力最集中的位置。
  • 去蕪存菁的高效餵食： 學習如何以最低的 token 消耗，精準傳遞最關鍵的邏輯脈絡，提升生成成功率。

📖 什麼是 Context Window？

LLM 讀文章跟人類很像，通常只記得最前面與最後面，中間的內容最容易被遺忘，這在學術上被稱為 Lost in the Middle 現象。

  • 工作記憶的侷限： 當 Context 超載時，AI 不僅容易遺忘規則，推理性能與指令遵循率也會跟著大幅滑落。
  • Lost in the Middle (中間遺忘) 效應：
    • 大模型對 Prompt 開頭與結尾的注意力最集中，中間的內容容易失焦。
    • 排版秘訣： 應將最關鍵的系統規則、生成限制與格式輸出要求放在 prompt 的最尾端。
  • 少即是多原則： 僅提供當前任務「剛剛好」的 Context。

💡 頓悟時刻

解決 Context 超載的終極手段不是寫更好的 Prompt，而是直接對 Context 進行「物理隔離」。

實體隔離（Physical Isolation）最安全：
配合下一堂將深入的 Subagent 機制，前端 Agent 的 Context 裡「絕對不會」出現後端的程式碼；反之亦然。

  • 物理隔離的好處：
    • 避免領域干擾： 刪除與當前任務無關的程式碼，AI 就不會無故改動不相干的邏輯。
    • 縮小搜索空間： 讓模型 100% 聚焦在目前要解決的單一功能模組上，最大程度消除幻覺。

🛠️ Token 效率最佳化技巧

不要給它整份實作，給它介面 (Interface)。這不僅省 Token 幫公司省錢，更能大幅提高 AI 生成的準確度。

  • Interface over Implementation (介面優先)：
    • 實戰範例： 貼上整份 2000 行的 class 原始碼，不如只貼上 50 行的介面 (Interface) 定義與公開 API 簽章。AI 只要知道怎麼調用，不需要知道裡面怎麼實作。
  • 結構化輸入 (Structured Input)： 使用 Markdown 階層 (如 `#` 標題、三個反單引號包覆的代碼區塊) 與 XML tags 劃分變數邊界，減少語意解析模糊。
  • 動態提取 Context： 結合 RAG（檢索增強生成）或開發工具，即時檢索與調用相關的程式片段，避免靜態塞入整份文件。

⚠️ 常見誤區

定期清理對話是工程開發中的好習慣。貪多嚼不爛，在 Context Engineering 中，「乾淨的對話」重於一切。

🤖 一鍵梭哈的陷阱

  • 將數百個檔案或上萬行程式碼直接貼進對話視窗。這樣做只會超出 Context 窗口，或導致嚴重的 "Lost in the Middle" 使得 AI 忽略你的特定指令。

⚙️ 忽視歷史包袱

  • 在一個對話中進行了數十次程式碼修改卻不重啟，舊的錯誤代碼與除錯脈絡仍在 Context 中持續干擾 AI 的新判斷。

💡 解決方法

開發好一個小模組後，請主動開啟新對話 (New Chat)，只把該模組的最終產出與介面作為新對話的背景輸入。
