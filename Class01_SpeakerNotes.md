# 第 1 堂：打造專屬助手：Agent Skills 的定義與運用
## 講者講稿 (Speaker Notes)

**課程總長：** 30 分鐘
**目標受眾：** Unity 前端與 .NET 後端開發人員
**設備需求：** 投影機、IDE (Visual Studio/Cursor)、已設定好的 AI 助手環境

開始之前，我先做個小調查。請問在座的各位，有誰在請 ChatGPT 或 Copilot 寫程式碼的時候，曾經拿到過看起來很完美，結果貼到專案裡卻狂噴編譯錯誤，或者完全不符合我們公司專案架構的程式碼？請舉手！

### [05:00 - 15:00] 核心概念解析 (Theory & Concepts)
**(Slide 2: 背景與核心痛點 & Slide 3: Agent Skills 的核心要素)**

**講者：**
「首先，我們要釐清兩個名詞：Agent 與 Skill。

如果說一般的 LLM 像是一本百科全書，那麼 Agent（代理人）就是一個具備行動能力的機器人。它可以讀你的檔案、跑你的編譯器。

那什麼是 Skill 呢？Skill 就是我們插在這個機器人背後的『技能卡匣』。

在我們的企業開發中，『通用解法』通常就是『錯誤解法』。一個好的 Skill，就是要填補這個斷層。它包含了四個要素：
1. **觸發條件**：什麼時候用它？
2. **上下文（Context）**：它需要知道專案的哪些資訊？
3. **約束條件（Constraints）**：這是最重要的！告訴它『絕對不能做什麼』。
4. **範例（Few-shot Examples）**：給它看滿分考卷長什麼樣子。

**(Slide 4: 頓悟時刻 Aha Moment)**
大家看到講義上那個發光的燈泡圖示了嗎？這是我希望你們今天帶走的最重要觀念：**Skill 不是 Prompt，它是『自動化工作流』的載體**。

把你的踩坑經驗寫成一個 Skill 存下來。下次新人進來，他只要呼叫這個 Skill，就能寫出跟 Senior 一樣水準的架構程式碼。這才是真正的 AI 賦能。」

### [15:00 - 25:00] 實戰演練與 Live Demo (Live Demo)
**(Slide 5: 實戰演練：.NET Repository Generator)**

**(切換畫面到 IDE，準備進行實際展示)**

**講者：**
「紙上談兵結束了，我們直接來看 Code。

各位後端夥伴，假設我們現在要寫一個取得使用者資料的 API。如果我直接請 AI 寫，（在 IDE 中輸入：『寫一個 C# 方法取得使用者資料』，展示 AI 給出直接操作 DbContext 的髒代碼）。

你看，它直接在 Controller 裡面撈資料庫。如果 Review 時看到這個，我一定會退件。

現在，我們換個作法。我們來建立一個名叫 `.NET Repository Generator` 的 Skill。

（展示講義中的 Markdown Skill 定義檔，重點解說 Constraints）

大家看這邊，我寫下了明確的『不准』：絕對禁止在 Controller 直接注入 DbContext。我還給了它一個 Golden Template（黃金範本）。

現在，我們載入這個 Skill，再問一次一模一樣的問題：『請幫我建立一個 User 的資料存取邏輯』。

（執行 AI 生成，展示完美符合 Repository Pattern 且有例外處理的程式碼）

Boom！看到了嗎？這就是 Skill 的威力。它不再是隨機盲猜，而是有紀律地工程輸出。」

### [25:00 - 30:00] 常見誤區與 Q&A 總結 (Wrap up & Q&A)
**(Slide 6: 常見誤區與防禦機制 & Slide 7: 常見問答 Q&A)**

**講者：**
「在我們自己嘗試寫 Skill 的時候，最容易犯的錯就是『太抽象』。不要對 AI 說『幫我寫好維護的程式』，你要說『每個 Function 不能超過 50 行』。AI 不懂哲學，它只懂規則。

另外，記得加上防呆機制。你可以在 Skill 的最後一行加上：『生成後，請自我檢查是否違反了上述的約束，若有，請重寫』。光是加上這一句話，就能濾掉 80% 的低級錯誤。

最後，回答大家最常問的問題：這值得花時間寫嗎？
絕對值得。你花 30 分鐘寫一個 Skill，整個團隊接下來一年在寫 CRUD 時，每個人每次省下 10 分鐘，還省下了無數次 Code Review 吵架的時間。

今天的課程就到這裡。下課後的作業是：找出你這個禮拜最常重複寫的一段 Boilerplate Code，嘗試為它寫一個專屬的 Skill。

我們下一堂課會深入探討 LLM 的底層運作邏輯，謝謝大家！」
