自動化測試你的 AI

實作 Evaluation Harness 與 CI/CD 整合

為何需要自動化測試？

當團隊規模擴大，如果不能量化 AI 的準確率，我們根本無法信任它。自動化測試是防範 AI 退化與回歸（Regression）的唯一手段。

「沒有量化準確率，AI 永遠只是個不穩定的玩具。」

🎯 自動化評測的核心價值：

  • 防範回歸 (Regression)： 避免 AI 出現「昨天會寫的功能，修改 Prompt 後今天突然不會了」的情況。
  • 量化指標： 提供明確的數據（如準確率高於 90%），作為 Prompt 或 Agent 能否合併上線的客觀依據。
  • 建立工程信任： 將不可預測的 LLM 黑盒子納入可量化、可重複測試的軟體工程規範中。

Harness 的三大要素

一個完整的 Evaluation Harness 就像是自動批次考試系統，由 Dataset、Runner 與 Evaluator 三者協同工作。

🤖 Dataset (資料集/考卷

包含多個測試案例。每個案例定義了輸入的 Prompt 以及預期的 Keyword 或程式碼結構。

⚙️ Runner (執行器/監考老師

負責批次調用 AI 介面，在沙盒環境下執行 Prompt 並蒐集產出的 Code。

💡 Evaluator (評分器/改考卷的

根據設定的規則（例如編譯通過與否、關鍵字包含等）進行自動化評估，若不達標則回傳 Error。

CI/CD 整合

將 Agent 參數與 Prompt 視為原始碼。透過發 PR 自動觸發 Harness 評測，防止劣質 Prompt 進入主線。

「Agent 設定檔如同程式碼，需納入 Git 嚴格版控。」

🤖 GitHub Actions 自動把關工作流：

  • 自動觸發測試： 每當 Prompt 變更或發送 PR 時，自動執行 Harness 測試集。
  • 不通過則阻擋： 若整體評測準確率低於門檻，直接阻擋 PR 合併，並在 PR 自動留言警示，防止越改越笨的 AI 參數進入主線。

📄 YAML 範例片段：

example.yaml
# 阻擋低準確率 PR
- name: Comment on PR
  if: failure()
  run: echo "❌ AI 準確率低於標準，阻擋合併！"

企業級 AI 四大守則

在企業環境中導入 AI，必須遵守四大黃金守則，確保安全性、可靠性與可維護性。

🛡️ 企業級開發守則：

  • 受測試檢驗： AI 只是特殊的編譯器，產出代碼必須一律通過既有的 Linter 與 Unit Test。
  • No CI, No AI： 沒有自動化評測腳本與持續整合把關，切勿在專案中大量採用 AI。
  • Prompt 版本控制： 所有的 System Prompt 與 Agent 設定必須納入 Git 版本控制，禁止手動在網頁後台修改。
  • 沙盒執行安全： AI 產出與測試執行必須在安全的沙盒環境進行，未經審閱不得直接在本地執行危險指令。

常見陷阱與防禦機制

實務執行自動化評測時，需防範過度擬合與測試成本高昂的問題，設計精準的防護機制。

⚠️ 常見挑戰與解法：

  • 過度優化 (Overfitting)： AI 容易針對已知考卷背答案。必須建立「盲測資料集（Blind Test Set）」，防止 Prompt 對特定測試過度擬合。
  • 裁判幻覺： LLM 裁判可能給出不穩定評分。建議以確定性測試（如編譯、單元測試、關鍵字匹配）為主，LLM 裁判僅作為風格風格輔助。
  • API 成本考量： 頻繁 PR 跑 API 可能花費不貲。實務上平常只跑 5~10 題的 Smoke Test，待 Release 前夕才跑完整的 Full Evaluation Harness。
