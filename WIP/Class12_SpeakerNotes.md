# 第 12 堂：自動化測試你的 AI：實作 Evaluation Harness 與 CI/CD 整合 - 講師講稿

**總時長：** 30 分鐘
**目標受眾：** 開發者、架構師、DevOps、QA

---

## [00:00 - 05:00] 開場與量化重要性
- **(投影片：為何需要自動化測試 AI？)**
- **講者台詞：**
  「大家好，這是我們系列的最後一堂課。
  大家都會寫 Prompt 了，但最大的問題是：昨天會寫的功能，今天 AI 突然不會了 (Regression)。在有 50 人的團隊中，如果不能量化 AI 的準確率，我們根本無法信任它。
  這堂課就是要教大家做 Evaluation Harness，讓 AI 成為工業級的可靠工具！」

## [05:00 - 15:00] Harness 實作解析
- **(投影片：Evaluation Harness 的三大要素)**
- **講者台詞：**
  「Harness 就像批次考試系統。包含資料集 (考卷)、執行器 (監考老師)、評分器 (改考卷的)。
  看講義的 JSON 範例，我們設定好要求包含哪些 Keyword。然後用 C# 寫一個 Runner，批次叫 Agent 去寫 Code，然後程式自動驗證結果。
  如果準確率低於 90%，我們就直接拋出 Error Code。這個動作是整合 CI 的關鍵！」

## [15:00 - 22:00] CI/CD 與 Git Flow
- **(投影片：CI/CD 與 Git Flow 整合)**
- **講者台詞：**
  「單機測試不夠。我們要把 Agent 的設定檔當作程式碼，納入 Git 版控。
  當你改了 Prompt 發 PR，GitHub Actions 自動觸發 Harness。如果分數不及格，CI 直接阻擋合併，並留言『準確率下降，請檢查！』
  這就用自動化機制，防堵了越改越笨的 AI 參數進入主線！」

## [22:00 - 27:00] 四大守則與陷阱防禦
- **(投影片：企業級 AI 開發四大守則)**
- **講者台詞：**
  「最後總結四大守則：
  第一，把 AI 當成特殊的編譯器，產出一樣要過 Lint 與 Unit Test。
  第二，No CI, No AI！沒有自動化測試把關，別大量用 AI。
  第三，所有 Prompt 必須進 Git 版控。
  第四，所有 Agent 產出的腳本只能在沙盒裡跑。
  另外小心過度優化 (Overfitting)，測試集要有盲測題，避免 AI 變成只會背答案。」

## [27:00 - 30:00] 總結與 Q&A
- **(投影片：結語與 Q&A)**
- **講者台詞：**
  「有人會問：『每次 PR 跑 API 很花錢？』實務上，平時只跑 5-10 題 Smoke Test，只有準備 Release 才跑 Full Harness。
  12 堂課到此告一段落！從 Prompt、Agent 到自動化評測，希望大家能把這套體系帶回團隊，建立最安全高效的 AI 工作流。謝謝大家！」
