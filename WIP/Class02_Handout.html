Agent 的本質與運作機制總結

📖 什麼是 Agent？

LLM 本身只是大腦，無法做任何事。Agent 就是給 LLM 賦予身體，在思考能力之上再加入感知與行動能力。

🤔 想一想，下列哪些算是 Agent 呢？

  • 案例1：給AI一篇文章，叫他總結文章重點。 (單純文字生成，不是 Agent)
  • 案例 2：在IDE裡使用代理人模式讓他直接讀寫程式碼。 (能呼叫系統工具，是基礎 Agent)
  • 案例 3：AI實作完自己執行Build與Test，自己發現錯誤並修正 (能在環境中感知與行動，是 Agent)
  • 案例 4：給予目標後，能自主規劃、建專案、寫程式碼、測試，不斷循環直到目標達成。 (具備規劃與反思修正能力，是高度自主的 Agent)
  • Chatbot (聊天機器人)： 接收輸入，靜態預測並輸出下一段文字，無法感知外部環境，也無法產生實際的副作用。
  • Agent (代理人)： 擁有明確的任務目標，能主動呼叫外部工具 (Tools)，並依據執行結果進行自我反思與修正。

Agent 的三大支柱

🧠 推理大腦 (Reasoner)

LLM 本身的推理與步驟規劃能力。

🛠️ 外部工具 (Tools)

讓 Agent 具備對系統、檔案與 API 的操控權限。

💡 反思修正 (Reflection)

讀取工具報錯並動態變更開發路線。

🎮 生動範例：指派 Junior 工程師修 Bug

把 Agent 想像成一位你剛帶的 Junior 工程師。你只要給他一個高階目標（例如：修復登入失敗的問題），他就會自主跑起一個解決問題的迴圈：

  • 👀 觀察 (Observation)：看 Log 尋找報錯。
  • 🧠 思考 (Thought)：推理錯誤原因，決定下一步。
  • 🛠️ 行動 (Action)：呼叫工具（讀取檔案、修改代碼、執行指令）。

這與傳統寫死規則的腳本完全不同，Agent 具備隨機應變的「容錯與反思」能力。

Agent_Trace_Log.txt
[User Prompt] 修復 Login 失敗的 Bug

[Action] 讀取 server.log
[Observation] 發現錯誤：Connection Timeout
[Thought] 這是資料庫連線超時，我該檢查 DB 設定檔。

[Action] 讀取 config.json
[Observation] 發現 DB_HOST 是 127.0.0.1
[Thought] 正式環境不該是 local IP，我來改成正確的環境變數。

[Action] 修改 config.json 並重啟 Server
[Observation] Server 成功啟動，登入正常。

[Thought] 任務完成，回報給主管！
[Action] 結束迴圈並回覆結果

💡 頓悟時刻

Agent 不是什麼高深莫測的魔法，它本質上就是一個掛了 LLM 推理大腦的 while 迴圈！

底層運作邏輯虛擬碼： 這解釋了為什麼它有時候會「想很久」，因為它正在背後瘋狂呼叫工具、讀檔、報錯、自我修正。
example.js
while (!task_completed) {
    let thought = think(goal, history);
    let action = select_tool(thought);
    let observation = execute(action);
    update_history(thought, action, observation);
    task_completed = evaluate(goal, history);
}

🛠️ 核心機制：ReAct 框架

Reasoning and Acting (ReAct) 是 Agent 最著名的運行框架。學會閱讀 Agent 的隱藏 Trace Log，是開發者最重要的除錯基本功。

🤖 Thought (思考

分析當前狀況與目標差距，推理下一步該做什麼。

⚙️ Action (行動

決定調用的工具 (Tool) 並傳入所需的引數 (Parameters)。

💡 Observation (觀察

工具執行後的實質反饋（如：檔案內容、編譯器報錯、命令執行結果）。

⚠️ 為什麼 Agent 會卡死？

當 Agent 陷入無效的重複思考或無限的 retry 迴圈時，我們必須介入並優化我們的指令與工具設計。

  • 工具不穩定與無防呆：
    • 工具噴出未處理的 Exception 或格式混亂的錯誤訊息，AI 無法解讀，只會以同樣的方式反覆嘗試，導致陷入死鎖。
  • 目標模糊 (Goal Ambiguity)：
    • 任務目標不夠具體（例如「幫我優化專案程式碼」），Agent 不知道什麼樣的狀態才算「完成」，因而無休止地運行下去。
  • 對策： 在提示詞中寫明任務完成的驗證條件（如「當執行 dotnet build 通過且無編譯錯誤時即代表任務完成」），並對工具加上明確的 Exception Handling。
