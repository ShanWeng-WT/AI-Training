# 第 11 堂：確保 AI 不退化：Harness Engineering 與評測基準 - 講師講稿

**總時長：** 30 分鐘
**目標對象：** Unity / .NET 開發人員、技術主管

---

## [00:00 - 05:00] 開場與痛點：AI 退化災難
- **(投影片：什麼是 AI 退化災難？)**
- **講者台詞：**
  「歡迎來到第 11 堂課。大家有沒有遇過，昨天 Prompt 跑得很完美，今天 AI 卻突然變笨、開始寫錯字或呼叫不存在的 API？
  這叫做『AI 退化』。如果我們不能量化、監控它的表現，我們就無法信任它。這堂課我們要學的就是幫 AI 寫『單元測試』，也就是 Harness Engineering。」

## [05:00 - 12:00] Harness Engineering 與 Ground Truth
- **(投影片：Harness Engineering 核心概念)**
- **講者台詞：**
  「Harness Engineering 的核心是建立 Ground Truth (黃金標準)。
  這包含三個東西：標準輸入的 Prompt、我們資深工程師寫好的標準答案、以及『評分標準 Rubrics』。這三點確立了，我們才有客觀的評量依據，不會憑感覺瞎猜 AI 的品質。」

## [12:00 - 20:00] 兩大評測方法論
- **(投影片：兩大評測方法論)**
- **講者台詞：**
  「有了標準後，怎麼打分？我們用兩把刷子。
  第一是『確定性評估』。最簡單暴力：能不能編譯？有沒有包含必要 API？這非常客觀。
  第二是『LLM-as-a-Judge』。既然它不管寫得優不優雅，我們就找 GPT-4o 來當裁判，評審架構、命名與註解，就像資深的 Tech Lead 在做 Code Review。」

## [20:00 - 26:00] 實作流程與實機 Demo
- **(投影片：實作流程概覽 / 切換終端機 Demo)**
- **講者台詞：**
  「我們看講義上的 JSON 檔，這就是考題，規定了要用到 `AddForce`，絕對不准用 `GameObject.Find`。
  *(執行 Python 評測腳本展示)* 
  大家看，我餵給系統一份爛 Code，確定性評估扣了分，LLM 裁判也精準抓出它在 Update 用 GetComponent 的效能問題，總分不及格。這就是自動化防護網！」

## [26:00 - 30:00] 陷阱與 Q&A
- **(投影片：常見陷阱與防禦)**
- **講者台詞：**
  「最後提醒，測試資料集要夠多元，不要發生過度擬合 (Overfitting)。還要小心 LLM 裁判的偏見，要在提示詞限制它不要只看字數長短。
  時間差不多，如果沒預算用大模型當裁判，用確定性評估把關編譯跟關鍵字，也能涵蓋 80% 的保護力。謝謝大家，下堂課見！」
