Agent Skills 的定義與運用

🎯 Skill 是什麼？ What is a Skill?

翻轉人機互動模式，將不可控的隨機對話，升級為品質穩定的「自動化技能」輸出。

😰 無技能的痛點

  • 重新探索成本每次執行相同任務，AI 都需重新推演解法，耗費額外時間與 Token。
  • 缺乏一致的標準AI每次解法與結論可能截然不同。
  • 品質難以掌握模型發散的輸出，導致難以預期與控管最終品質。

💡 核心解決方案

  • 固化專案知識在 SKILL.md 中載明團隊固定的開發知識、指令與標準作法。
  • 可執行的工作流透過內建的程式腳本、明定的指令，讓 AI 按既定邏輯執行，確保產出一致。
  • 賦予行為邊界不再依賴單次對話，改用系統化定義來約束 AI 行為。

🎯 達成的目標

  • 穩定且精準產出結果變得可預測且高度一致。
  • 經驗資產化將踩坑經驗與最佳實踐轉化為隨插即用的「技能」。
  • 提升效益比杜絕反覆溝通與無效試錯，節省時間及Token。

⚙️ Agent Skill 的 標準與結構

每一個 Skill 都必須遵循特定的目錄結構與命名規範，才能被系統正確載入並辨識。

  • 命名規範：目錄名稱與 name 屬性必須完全一致，建議使用 kebab-case (小寫與連字號，如 my-skill)。
  • SKILL.md (必填)：技能的核心定義檔，上方必須包含 YAML Frontmatter 宣告。
  • 其他目錄 (選填)：如 scripts/ 放置腳本、references/ 放置參考文件、assets/ 放置模板檔案。
Directory & SKILL.md
your-skill-name/ 
├── SKILL.md        # Required - main skill file 
├── scripts/        # Optional - executable code 
├── references/     # Optional - documentation 
└── assets/         # Optional - templates, etc.

# SKILL.md 檔案內容開頭必須包含：
---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---

💡 頓悟時刻 Aha Moment

Skill 不是 Prompt！它是可重複使用的「知識軟體化」與「自動化工作流」。

"Skill 不是隨機的文字對話！它是「可重複使用的自動化工作流」的載體。"

將個人或團隊的踩坑經驗編寫成 Skill 存檔，其帶來的核心價值包括：

  • 知識軟體化： 把資深工程師的架構思維與開發規範，轉化為 AI 可讀的指令集。
  • 經驗無縫傳承： 新進人員只需呼叫相同的 Skill，即可瞬間產出與團隊一致的高品質代碼。
  • 自動化防線： 減少手動調整 Boilerplate 的時間，將精力集中在核心業務邏輯。

🛠️ 實戰： .NET Repository Generator

展示如何透過 Constraints 與黃金範本，規範 AI 產出符合架構的 CRUD 邏輯。

實際痛點： 直接發問會導致 AI 在 Controller 注入 DbContext 並直接操作，違反架構分層。

解決方案： 建立 Repository 產生技能，以 Markdown 明確定義規則：

※ 套用此技能後，AI 即可產出符合團隊分層設計與例外處理的代碼。

example.md
# Constraints (約束條件):
1. 絕對禁止在 Controller 直接注入與使用 DbContext。
2. 必須建立 `I[Entity]Repository` 介面，並在 `[Entity]Repository` 實作。
3. 所有資料存取方法必須使用 `async Task<T>` 異步設計。
4. 必須包含 Try-Catch 例外處理並記錄 Log。

⚠️ 常見誤區與 防禦機制

AI 不懂哲學，它只懂明確的規則。設計 Skill時應避免空泛指令，並建立自我檢視機制。

🤖 空泛指令

避免使用「寫出好維護的程式」，改用具體量化指標（例如：每個方法不可超過 50 行，圈複雜度 < 10）。

⚙️ 缺乏上下文範圍

避免無範圍生成，應精確限制範圍（例如：僅限 .NET 8 / EF Core，或特定 Unity UI Toolkit 框架）。

💡 無防呆與自我修正

增加自我檢查提示詞（例如：在 Skill 最後要求「生成代碼後，請對照 Constraints 自我檢查，若有違反請重新生成」），能有效濾掉 80% 的低級錯誤。

❓ 常見問答 Q&A

面對開發團隊對 Skill 的實用性、成本與維護性的常見質疑。

❌ Q1: 每次寫 Skill 都要花時間，會不會反而拖慢開發進度？


A1: 這是高投報率的長期投資。花 30 分鐘寫一個 Skill，能讓整個團隊在往後的一整年裡，每次生成 Boilerplate 省下 10 分鐘，同時省去無數 Code Review 的糾錯與爭吵時間。

🎯 Q2: 把一堆 Constraints 和範本塞進 Prompt，會不會導致 Token 爆掉或 AI 遺忘？


A2: 應採用動態上下文與限縮範疇。將 Skill 切割成單一職責（Single Responsibility），只在對應的場景載入對應的 Skill，並使用 RAG 或編輯器 context 功能來精準控制 Token 數量。
