打造專屬助手：
Agent Skills 的定義與運用

企業內部 AI 輔助開發教育訓練

🎯 課程目標
Session Goals

翻轉人機互動模式，從隨機對話升級為系統化的「自動化技能」輸出，真正解決開發痛點。

本堂課將帶領大家將個人的開發經驗軟體化，建立第一個專屬 Agent Skill：

  • 建立技能： 實務操作 Agent Skills，將踩坑經驗轉化為可複用的代碼生成規則。
  • 翻轉模式： 擺脫每次對話重新說明的困境，改以系統化技能卡匣來約束並引導 AI。
  • 解決痛點： 杜絕 AI 瞎編程式碼、不符合團隊架構的亂象，使開發效率與代碼品質大躍進。
目標：把一般的 LLM（百科全書）改造成能直接融入 Wanin 開發工作流的專屬助手。

📖 背景與
核心痛點

通用型 AI 缺乏專案感知，反覆溝通與糾錯的時間成本反而拖慢了開發效率。

當我們把 AI 當成「聰明但無記憶的大學實習生」時，會面臨以下痛點：

  • 通用解法就是錯誤解法： AI 不懂特定的專案架構、程式庫規範與企業內部標準。
  • 同步成本極高： 每次開啟新對話都要重述一遍專案規範，稍有不慎即產生編譯錯誤。
  • 解決方案： 透過 Agent Skills 賦予 AI 專案感知力與行為邊界，省下反覆 Debug 與 Code Review 的爭執時間。

⚙️ Agent Skills 的
核心要素

技能卡匣的四大核心支柱，決定了 AI 生成結果的紀律與精準度。

一個設計良好的 Skill（技能定義檔）通常包含以下幾大區塊：

  • Intent (觸發條件)： 明確定義什麼時候該使用此技能（例如：看見實體類別時生成 Repository）。
  • Context (專案上下文)： 提供 AI 必須理解的專案環境（框架版本、特定庫或資料庫結構）。
  • Constraints (約束條件)： 最核心的部分，明確列出「絕對禁止的事情」（例如：不可在 Controller 使用 DbContext）。
  • Few-shot (黃金範本)： 給予一到多個標準的滿分考卷範例，讓 AI 依葫蘆畫瓢。
  • Next Actions (後續流程)： 定義代碼生成後的下一步驗證或說明流程。

💡 頓悟時刻
Aha Moment

Skill 不是 Prompt！它是可重複使用的「知識軟體化」與「自動化工作流」。

"Skill 不是隨機的文字對話！它是「可重複使用的自動化工作流」的載體。"

將個人或團隊的踩坑經驗編寫成 Skill 存檔，其帶來的核心價值包括：

  • 知識軟體化： 把資深工程師的架構思維與開發規範，轉化為 AI 可讀的指令集。
  • 經驗無縫傳承： 新進人員只需呼叫相同的 Skill，即可瞬間產出與團隊一致的高品質代碼。
  • 自動化防線： 減少手動調整 Boilerplate 的時間，將精力集中在核心業務邏輯。

🛠️ 實戰：
.NET Repository Generator

展示如何透過 Constraints 與黃金範本，規範 AI 產出符合架構的 CRUD 邏輯。

實際痛點： 直接發問會導致 AI 在 Controller 注入 DbContext 並直接操作，違反架構分層。

解決方案： 建立 Repository 產生技能，以 Markdown 明確定義規則：

# Constraints (約束條件):
1. 絕對禁止在 Controller 直接注入與使用 DbContext。
2. 必須建立 `I[Entity]Repository` 介面，並在 `[Entity]Repository` 實作。
3. 所有資料存取方法必須使用 `async Task<T>` 異步設計。
4. 必須包含 Try-Catch 例外處理並記錄 Log。

※ 套用此技能後，AI 即可產出符合團隊分層設計與例外處理的代碼。

⚠️ 常見誤區與
防禦機制

AI 不懂哲學，它只懂明確的規則。設計 Skill時應避免空泛指令，並建立自我檢視機制。

設計 Agent Skills 時常見的三大誤區與改進方式：

  • 空泛指令： 避免使用「寫出好維護的程式」，改用具體量化指標（例如：每個方法不可超過 50 行，圈複雜度 < 10）。
  • 缺乏上下文範圍： 避免無範圍生成，應精確限制範圍（例如：僅限 .NET 8 / EF Core，或特定 Unity UI Toolkit 框架）。
  • 無防呆與自我修正： 增加自我檢查提示詞（例如：在 Skill 最後要求「生成代碼後，請對照 Constraints 自我檢查，若有違反請重新生成」），能有效濾掉 80% 的低級錯誤。

❓ 常見問答
Q&A

面對開發團隊對 Skill 的實用性、成本與維護性的常見質疑。

  • Q1: 每次寫 Skill 都要花時間，會不會反而拖慢開發進度？
    A1: 這是高投報率的長期投資。花 30 分鐘寫一個 Skill，能讓整個團隊在往後的一整年裡，每次生成 Boilerplate 省下 10 分鐘，同時省去無數 Code Review 的糾錯與爭吵時間。
  • Q2: 把一堆 Constraints 和範本塞進 Prompt，會不會導致 Token 爆掉或 AI 遺忘？
    A2: 應採用動態上下文與限縮範疇。將 Skill 切割成單一職責（Single Responsibility），只在對應的場景載入對應的 Skill，並使用 RAG 或編輯器 context 功能來精準控制 Token 數量。
